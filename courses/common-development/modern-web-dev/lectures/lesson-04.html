SECTION 01

요청 데이터의 위치

HTTP 요청에는 메서드, 경로, 헤더, 본문이 있다. Express에서는 이 요청을 `req` 객체로 받고, 응답을 `res` 객체로 만든다. 강의는 실습을 통해 `req.query`, `req.params`, `req.body`를 차례로 확인한다.

실습의 공통 준비 코드는 화면처럼 JSON 본문, 정적 파일, URL-encoded 폼 본문을 읽을 수 있는 미들웨어를 먼저 등록한다. 이 세 줄이 있어야 이후 라우트에서 `req.body`와 정적 파일 제공이 의도대로 동작한다.

const express = require("express");
const app = express();

app.use(express.json());

app.use(express.urlencoded({ extended: true }));

// ...

SECTION 02

query와 params

 
app.use(express.urlencoded({ extended: true }));
app.use(express.json());

app.get("/search", (req, res) => {
  console.log(req.query.keyword);
  res.send(`keyword = ${req.query.keyword}`);
});

app.get("/users/:userId", (req, res) => {
  console.log(req.params.userId);
  res.send(`user id = ${req.params.userId}`);
});

app.post("/users", (req, res) => {
  console.log(req.body);
  res.json({ ok: true, received: req.body });
});

`req.query`는 URL의 `?` 뒤에 붙은 키-값 쌍을 객체처럼 읽는다. 검색어, 필터, 페이지 번호처럼 리소스 조회 조건에 자주 사용된다. `req.params`는 라우트 경로에 `:userId`처럼 선언한 동적 구간을 읽는다. 사용자의 상세 페이지, 게시글 상세 페이지처럼 특정 리소스를 가리킬 때 많이 쓴다.

SECTION 03

body와 파싱 미들웨어

`req.body`는 POST, PUT, PATCH 요청처럼 본문에 담긴 데이터를 읽을 때 사용한다. 단, Express가 본문을 자동으로 항상 해석해 주는 것은 아니다. 폼 데이터는 `express.urlencoded`, JSON 데이터는 `express.json` 미들웨어가 먼저 등록되어야 한다.

보안 관점에서는 `req.body`를 읽는 순간부터 사용자가 보낸 값이 서버 로직 안으로 들어온다. 따라서 길이, 타입, 허용 값, 필수 값, 특수 문자, 권한을 검증하지 않으면 인젝션, 권한 우회, 서버 오류로 이어질 수 있다.

아 그러면 저희는 여기서 알아낼 수 있죠. 아 리퀘스트 객체의 query라는 값이 요렇게 query string을 받는 값이구나 아 방법이구나 해서 콘솔 로그로 찍어 볼게요. 다시 query를 콘솔 로그 찍게 하고 리퀘스트를 주면 q는 1, 2, 3이 뜨겠고 q에 이제 q 값을 또 찍어 주면 여기서 이제 1, 2, 3 값이 이렇게 정상적으로 출력이 되겠죠. 이거를 뭐 예를 들어서 q가 아니라 뭐 유저네임 이런 식으로 받아서 if request query 유저네임이 뭐 어드민이면 콘솔 아니 뭐 res send를 hello admin 이라고 줘라 이런 로직을 또 구현을 할 수 있을 거예요. 그죠? 당연히 굉장히 보안성으로 취약한 코드지만 여기에 query string을 유저네임에 어드민을 주게 되면 hello admin이라는 이 조건문을 통과해서 이렇게 query string을 다뤄볼 수 있습니다. 네 바로 다음으로 두 번째 실습 진행해 보도록 하겠습니다. 좀 전에 이제 query string을 다루는 방법을 배웠는데요. 이번에는 리퀘스트에 있는 다른 그 속성인 params에 대해서 배워 볼 거예요.

이번엔 query가 아니라 params에 대해서 배워 볼 건데 뭐 예를 들면 이런 게 있어요. 아마 REST API에 대해서 혹시 배워 보신 분들이 있으실 텐데 있으실 수도 있는데 어 좀 뭐 방법론 중에 하나거든요. 웹 애플리케이션을 개발하기 위한 방법론 중 하나 중 뭐 예를 들어서 users라는 라우팅에 get 메소드를 주면 유저의 데이터를 가져오고 post 하면은 유저의 뭔가 데이터를 주고 put 하면 유저를 추가하고 delete 하면 유저를 삭제하는 이런 방법론이 있어요. 이때 많이 사용을 하는데 예를 들어서 이런 식으로 REST 풀 하게 구현을 하면 여기에 뭐 delete 1 하면서 유저의 1번 인덱스를 가진 유저를 삭제한다 뭐 2번 유저를 추가한다 3번 유저의 데이터에 뭘 추가한다 뭐 10번 유저의 데이터를 가져온다 이런 식으로 구현하는 걸 이제 REST 풀 하게 구현을 한다 라고 하는데 REST 풀 API 이게 요즘 되게 많이 사용하고 있는 웹 앱 개발 패러다임인데 요거를 사용을 하려면 이제 요. 값을 받아야 되잖아요.

요. 값을 받는 방법이 바로 params입니다. 빠르게 실습을 한번 같이 해볼게요. params를 받는 방법은 똑같이 이제 get 메소드로 한번 예습을 받아 볼게요. 뭐 예를 들어서 유저라는 라우터에 제가 뒤에 있는 id 값을 받아 본다고 해볼게요. 뭐 예를 들어서 유저 뒤에 인덱스가 뭐 13번 유저를 가져온다 라고 예를 들어 봤을 때 여기에 13을 가져야 되잖아요. 근데 예를 들어서 유저가 100명이 있다고 하면 저희가 뭐 이거를 뭐 이렇게 100개를 만들 순 없잖아요. 이렇게 1, 2, 3, 4 이렇게 100개를 만들 순 없으니까 이거를 동적으로 받긴 해야 되는데 이걸 받는 방법은 바로 이렇게 받습니다. 이 콜론을 주고 id를 이런 식으로 주면 그리고 뒤에 콜백함수 연해주고 여기서 이제 콘솔로그 리퀘스트 한번 받아 볼게요. 자 서버 다시 실행시켜주고 여기서 이제 유저의 뒤에 뭐 예를 들어서 13을 넣어주면 지금 행이 걸려 있는데 이 행이 걸려 있는 이유는 우리가 리스폰스를 안 줬죠. 그렇기 때문인데 제가 서버에서는 아마 정상적으로 콘솔로그가 실행이 됐을 겁니다.
