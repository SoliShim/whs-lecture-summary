SECTION 01

Node.js의 위치

강의는 Node.js를 “브라우저 밖에서도 JavaScript를 실행할 수 있게 해 주는 런타임”으로 설명한다. 원래 JavaScript는 웹 브라우저에서 동작하는 언어로 출발했지만, Node.js는 Chrome V8 엔진을 기반으로 서버, 명령행 도구, 자동화 도구, 웹 애플리케이션 백엔드에서 JavaScript를 실행할 수 있게 만든다.

웹 보안 관점에서 Node.js를 공부하는 이유는 Express, EJS, npm 패키지, 서버 라우팅, 미들웨어, 템플릿 렌더링 같은 실제 공격 표면이 Node.js 생태계에서 자주 등장하기 때문이다. 이 과목은 단순히 문법을 배우는 것이 아니라 이후 Prototype Pollution과 EJS RCE를 이해하기 위한 기반을 쌓는 흐름이다.

SECTION 02

주요 특징

Node.js의 핵심 특징은 비동기 논블로킹 I/O, 싱글 스레드, 이벤트 기반 구조다. I/O는 파일 읽기, 네트워크 요청, 데이터베이스 접근처럼 외부 자원과 데이터를 주고받는 작업을 말한다. 서버 프로그램은 CPU 계산보다 I/O 대기가 많은 경우가 많기 때문에, 대기 시간 동안 다른 요청을 처리할 수 있으면 효율이 높아진다.

특징의미주의점
비동기작업을 요청한 뒤 완료를 기다리는 동안 다음 작업을 진행한다.콜백, Promise, async/await 흐름을 읽을 수 있어야 한다.
논블로킹 I/O입출력 완료 전까지 실행 흐름 전체를 멈추지 않는다.CPU 계산이 길면 이벤트 루프를 막을 수 있다.
싱글 스레드JavaScript 실행은 주로 하나의 스레드에서 순차적으로 처리된다.무거운 연산은 워커나 별도 프로세스로 분리해야 한다.
이벤트 기반요청, 응답, 파일 완료 같은 사건이 발생하면 등록된 함수가 실행된다.이벤트 흐름을 놓치면 디버깅이 어려워진다.
SECTION 03

동기와 비동기

동기 처리에서는 Job A가 끝나야 Job B를 시작하고, Job B가 끝나야 Job C를 시작한다. 실행 순서를 이해하기 쉽지만, 어떤 작업이 오래 걸리면 뒤의 모든 작업이 기다린다. 반대로 비동기 처리에서는 오래 걸리는 작업을 요청해 둔 뒤 다음 작업을 먼저 진행하고, 완료 시점에 결과를 받아 처리한다.

서버는 동시에 많은 클라이언트의 요청을 받는다. 요청마다 파일, DB, 외부 API 응답을 기다리는 시간이 생긴다. Node.js는 이 대기 시간에 다른 요청 처리를 이어갈 수 있어 네트워크 중심 애플리케이션에 적합하다.

SECTION 04

블로킹과 논블로킹

동기/비동기는 작업 완료 통지를 어떻게 다루는지에 가깝고, 블로킹/논블로킹은 호출한 쪽의 실행 흐름이 멈추는지에 가깝다. 블로킹 호출은 결과가 나올 때까지 다음 줄로 넘어가지 않는다. 논블로킹 호출은 요청만 전달하고 즉시 제어권을 돌려받아 다른 일을 할 수 있다.

강의는 이 개념을 서버 성능과 연결한다. 모든 요청이 블로킹으로 처리되면 느린 파일 읽기나 네트워크 응답 하나가 전체 처리량을 떨어뜨린다. Node.js의 장점은 이벤트 루프와 비동기 API를 통해 이런 대기를 효율적으로 다루는 데 있다.

SECTION 05

싱글 스레드와 이벤트 기반

Node.js가 싱글 스레드라는 말은 모든 일을 하나의 줄에서만 처리한다는 뜻이 아니다. JavaScript 코드 실행은 한 줄기 흐름으로 관리되지만, 파일 시스템·네트워크 같은 I/O는 내부적으로 운영체제와 libuv의 도움을 받아 처리되고 완료 이벤트가 다시 JavaScript 영역으로 전달된다.

장점은 구조가 가볍고 많은 I/O 요청을 효율적으로 받을 수 있다는 점이다. 단점은 CPU를 오래 붙잡는 계산, 무한 루프, 무거운 암호 연산 같은 작업이 이벤트 루프를 막으면 전체 응답성이 떨어진다는 점이다. 따라서 서버 코드는 “짧게 실행하고, 오래 기다리는 일은 비동기로 맡긴다”는 원칙으로 작성해야 한다.

한 작업이 끝날 때까지 다음 작업을 시작하지 않습니다. 예를 들면 저희가 이 풀트럭에서 줄을 서서 차례대로 주문을 하는 거랑 비슷하다고 보면 될 것 같아요. 이 반대로 비동기, Synchronous라고 하는데 이 비동기는 여러 작업에 동시 처리가 가능한 것을 의미합니다. 동기와 다르게 한 작업이 끝나기를 기다리지 않고 다른 작업을 시작할 수가 있습니다. 예를 들면 저희 패스트푸드점에서 주문을 하고 저희 진동배를 받고 기다리는 동안 휴대폰을 볼 수도 있고 다른 일을 할 수 있는 것과 비슷하다고 생각하시면 될 것 같아요. 좀 더 쉽게 설명하기 위해서 좀 원론적인 설명을 해볼게요. 작업 A의 종료시간과 작업 B의 시작시간이 같으면 동기적이라고 합니다. 호출된 작업의 리턴 시간과 결과를 반환하는 시간이 일치하지 않는 경우를 말하는데요. 동기와 반대로 작업 A의 종료시간과 B의 시작시간이 같지 않을 때 저희는 비동기적이라고 봅니다. 극단적인 예를 한번 들어볼게요. 작업 A가 10초 작업 B가 2초 걸리는 작업인데 이 B 작업이 끝나는데 적어도 10초를 기다려야 하는 경우에는 동기적이 되겠고요.
