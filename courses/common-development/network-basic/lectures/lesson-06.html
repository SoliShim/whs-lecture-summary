SECTION 01

응용 계층까지 올라오기

강의는 지금까지 배운 계층을 다시 복습하며 시작한다. 데이터 링크 계층은 근거리 네트워크에서 프레임을 이웃 네트워크 요소로 전달한다. 네트워크 계층은 호스트 대 호스트 전달과 라우팅을 담당하고, IP 프로토콜과 데이터그램을 다룬다. 전송 계층은 TCP와 UDP를 통해 프로세스 대 프로세스 논리적 연결을 가능하게 한다.

응용 계층은 이 하위 계층들의 서비스를 기반으로 실제 네트워크 애플리케이션의 메시지를 다룬다. 강의는 다시 한 번 각 상위 계층이 하위 계층 서비스와 기능에 의존한다고 강조한다.

SECTION 02

네트워크 애플리케이션 구조

네트워크 애플리케이션은 네트워크를 사용해 통신하는 모든 프로그램이다. 강의에서는 크게 클라이언트-서버 구조와 P2P 구조로 나누어 설명한다.

구조설명예시·특징
클라이언트-서버서버가 있고 여러 클라이언트가 서버에 접근한다.웹사이트 대부분. 서버는 고정 IP 주소가 필요하다.
P2P피어라고 부르는 호스트 쌍들이 서버를 거치지 않고 직접 통신한다.토렌트가 대표 예시다. 비용 효율적이고 자기 확장성이 있다.

클라이언트-서버 구조를 이해하는 것은 이후 클라이언트 사이드 공격과 서버 사이드 공격을 이해하는 데 큰 도움이 된다. 어떤 요청이 클라이언트에서 시작되고, 서버가 어떤 응답을 주며, 취약점이 어느 쪽 로직에서 발생하는지 구분할 수 있기 때문이다.

SECTION 03

프로세스 통신과 소켓

클라이언트 프로세스와 서버 프로세스는 컴퓨터 네트워크를 통해 응용 계층 메시지를 교환한다. 이 통신을 가능하게 하는 메커니즘은 앞 강의에서 배운 전송 계층이다. 각 클라이언트 애플리케이션과 서버 애플리케이션에는 소켓이라는 인터페이스가 있고, 이 소켓은 애플리케이션 계층과 전송 계층 사이에서 통신 출입구 역할을 한다.

따라서 응용 계층을 이해할 때도 소켓과 포트 번호를 함께 떠올려야 한다. 브라우저가 HTTP 메시지를 만든다고 해서 브라우저 혼자 네트워크 전체를 처리하는 것이 아니라, 소켓을 통해 전송 계층과 연결되고 그 아래 계층들의 서비스를 타고 실제 네트워크로 나간다.

SECTION 04

HTTP의 특성

HTTP는 웹 프로토콜이며, 현대 웹 사용 경험의 중심이다. 뉴스, 게임, 영화, Netflix, Discord 같은 여러 서비스가 웹 기술 위에서 동작한다. HTTP를 이해해야 이후 웹 해킹과 네트워크 패킷 분석을 제대로 이해할 수 있다.

강의는 HTTP의 중요한 특성으로 무상태성과 비연결성을 설명한다. HTTP 서버는 기본적으로 두 요청 사이의 상태를 유지하지 않는다. 그런데 우리가 로그인 상태를 유지할 수 있는 이유는 HTTP 자체가 상태를 기억하기 때문이 아니라, 쿠키와 서버 측 세션 같은 메커니즘이 추가로 사용되기 때문이다. 또한 HTTP는 하나의 요청에 하나의 응답이 오는 구조로 설명된다.

HTTP는 기본적으로 상태를 기억하지 않는다. 로그인 유지, 장바구니, 사용자 식별은 쿠키와 세션 같은 별도 장치가 붙어서 가능해진다.
SECTION 05

HTTP request

HTTP request는 클라이언트가 서버에 어떤 작업을 요청하는 메시지다. 첫 줄에는 method, path, HTTP version이 들어간다. 그 아래에는 Host, User-Agent, Accept 같은 여러 header가 오고, 필요하면 빈 줄 뒤에 body가 붙는다.

구성 요소의미
methodGET, POST처럼 어떤 방식의 요청인지 나타낸다.
path서버 안의 어떤 경로에 접근할지 나타낸다.
versionHTTP/1.1 같은 프로토콜 버전을 나타낸다.
Host어떤 서버 주소로 요청을 보낼지 나타낸다.
header요청에 대한 부가 정보를 key-value 형태로 담는다.
bodyPOST 요청처럼 서버에 보낼 데이터를 담을 수 있다.

GET은 URL path 뒤의 query 부분에 매개변수를 담아 보내는 경우가 많고, POST는 HTTP request body에 값을 담아 보낸다. 강의는 URL을 브라우저에 입력해 들어가는 동작을 GET으로, 로그인 폼에 아이디와 비밀번호를 넣어 보내는 동작을 POST 예시로 설명한다.

SECTION 06

HTTP response

HTTP response는 서버가 클라이언트에게 보내는 응답 메시지다. request와 비슷하게 header와 body를 가지지만, 첫 줄은 상태 줄이다. 상태 줄에는 HTTP version, status code, status text가 들어간다.

상태 코드의미
200 OK정상적으로 처리되어 응답이 왔다는 뜻이다.
404 Not Found서버에서 해당 path를 찾을 수 없다는 뜻이다.
500 Internal Server Error서버 내부 오류가 발생했다는 뜻이다.

응답 body에는 HTML 같은 클라이언트에게 보여 줄 데이터가 들어갈 수 있다. 브라우저는 이 응답 본문을 읽어 사용자가 보는 웹페이지로 렌더링한다. 그래서 HTTP를 이해하면 브라우저가 서버와 어떻게 대화하고 화면을 어떻게 구성하는지 더 잘 이해할 수 있다.

SECTION 07

HTTP, HTTPS, 보안

강의는 HTTP가 네트워크 관점에서 취약하다고 설명한다. MITM 공격자가 중간에서 패킷을 sniffing하면 HTTP 평문이 그대로 노출될 수 있다. 만약 로그인 과정에서 아이디와 비밀번호가 HTTP 평문으로 오간다면 공격자가 그 정보를 볼 수 있다.

HTTPS는 HTTP에 secure가 붙은 형태로 설명된다. 기본적으로 HTTP를 기반으로 하지만 통신 내용이 암호화되어 있어, sniffing을 하더라도 안에 담긴 정보를 쉽게 알아내기 어렵다. 따라서 기밀성을 보장할 수 있다. 강의는 해외 사이트와 국내 대형 사이트 대부분이 HTTPS를 적용하고 있지만, 아직 적용되지 않은 사이트도 존재한다고 덧붙인다.

SECTION 08

DNS

응용 계층 프로토콜은 HTTP만 있는 것이 아니다. DNS, SMTP 같은 여러 프로토콜이 있다. 이 강의에서는 HTTP 다음으로 DNS를 소개한다. 우리가 웹사이트에 접속할 때 보통 IP 주소를 직접 입력하지 않고 naver.com 같은 도메인 이름을 입력한다. DNS는 이런 호스트 이름을 IP 주소로 변환해 주는 서비스다.

브라우저가 어떤 홈페이지에 접근하려 하면 먼저 DNS query를 수행해 도메인 이름에 대응되는 IP 주소를 찾는다. 그 IP 주소와 포트를 기반으로 웹 서버에 연결하고 response를 가져온다. 강의는 DNS를 중앙집중형 데이터베이스라고 설명하면서도, 한 서버가 고장나거나 트래픽 과부하로 망가지면 문제가 생길 수 있기 때문에 많은 서버를 계층 형태로 구성해 전 세계에 분산해 두었다고 설명한다.

SECTION 09

TCP/IP 모델과 OSI 7계층

강의 마무리에서는 TCP/IP 계층 구조를 다시 강조한다. 물리, 데이터 링크, 네트워크, 전송, 응용 계층의 역할과 단위를 꼭 기억해야 한다. OSI 7계층도 함께 많이 언급되지만, 멘토는 패킷 분석과 프로그래밍에서 직접적으로 다루기 좋은 TCP/IP 레이어를 기준으로 설명했다고 말한다. 더 궁금하면 자료 조사나 책을 통해 OSI 7계층도 추가로 읽어볼 수 있다.

챕터 6 애플리케이션 계층 내용 시작하도록 하겠습니다. 저희가 지금까지 네트워크 구조 모델에 대해서 계층 구조를 한번 알아보았고 지금까지 데이터 링크, 네트워크 레이어 그리고 트랜스포트 레이어까지 저희가 공부를 했습니다. 사실 지금 사계층까지 공부를 했고 마지막 남은 계층이 애플리케이션 계층인데요. 다시 한번 그 계층들이 어떻게 이루어져 있고 단위가 어떤 것인지 이해를 해보시는 게 좋을 것 같습니다. 데이터 링크 레이어는 한 네트워크 요소부터 이온 네트워크 요소로 프레임이라는 단위를 전달하는 근거리 네트워크에서 프레임이라는 단위를 전달하는 역할을 한다 했고요. 네트워크 레이어는 호스트 대 호스트로 어떤 값을 보낼 때 라우팅을 결정짓고 어떻게 호스트 대 호스트로 전달을 할지 IP 프로토콜 등에 대한 얘기를 한번 나눠봤습니다. 그리고 전 강의였던 트랜스포트 계층에서는 그 어플리케이션의 프로세스 대 프로세스 대 논리적 연결을 가능하게 하는 TCP, UDP와 같은 프로토콜을 다뤘었고요. 다음은 이것들을 기반으로 어플리케이션 계층에서는 어떤 프로토콜이 이루어지는지 이해를 한번 해보겠습니다.
