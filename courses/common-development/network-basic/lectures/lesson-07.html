학습 목표

  • Wireshark의 목적과 공격·방어 관점의 활용을 이해한다.

  • Follow TCP Stream, ip.addr 필터, Flow Graph로 TCP 3-way handshake를 확인한다.

Wireshark를 쓰는 이유

이 강의는 지금까지 배운 TCP/IP 5계층이 실제 패킷에서 어떻게 보이는지 확인하는 실습이다. 멘토는 Wireshark를 직접 설치하고 따라 하면서 이해하는 것을 권장한다. 패킷이 실제로 어떻게 오가고 어떤 계층 구조를 가지는지 보는 능력은 네트워크 프로그래밍, 취약점 분석, BoB와 같은 보안 학습에서 중요한 기반이 된다.

Wireshark는 호스트 장치에서 발생하는 네트워크 패킷을 캡처하고 분석할 수 있는 도구다. 공격자 관점에서는 공격 이후 sniffing에 사용할 수도 있고, 방어자 관점에서는 침해사고 대응, 트래픽 분석, 포렌식에 활용할 수 있다. 오프라인 강의에서는 패킷을 직접 다루고 다양한 공격까지 구현해 볼 계획이라고 설명한다.

설치와 인터페이스 선택

Wireshark는 공식 홈페이지에서 Windows, macOS, Linux 버전을 내려받아 설치할 수 있다. 멘토는 macOS ARM 버전을 사용한다고 설명한다. 실행하면 어떤 네트워크 인터페이스를 캡처할지 선택하는 화면이 나온다. 강의에서는 Wi-Fi로 연결되어 있기 때문에 Wi-Fi 네트워크 인터페이스를 선택해 캡처를 진행한다.

캡처를 시작하면 TCP, QUIC 등 수많은 프로토콜 패킷이 계속 오가는 것을 볼 수 있다. Chrome 기반 브라우저에서는 QUIC도 계속 보일 수 있다. 그래서 특정 실습에서는 필요한 패킷만 보기 위해 display filter를 사용해야 한다.

HTTP 사이트 접속과 필터

실습 목표는 HTTP 웹사이트에 접속했을 때 request와 response가 어떻게 보이는지 확인하는 것이다. HTTPS가 적용된 사이트는 내용이 암호화되어 HTTP 본문을 쉽게 확인하기 어렵기 때문에, 강의에서는 HTTPS를 사용하지 않는 개인 HTTP 서버를 예시로 든다. 학생도 HTTPS가 적용되지 않은 테스트용 HTTP 사이트를 찾아 실습할 수 있다고 설명한다.

흐름은 간단하다. Wireshark 캡처를 시작하고, 브라우저로 HTTP 서버에 접속한 뒤, 캡처를 멈춘다. 그 다음 display filter에 http를 입력하면 HTTP 통신이 이루어진 패킷만 볼 수 있다. 강의에서는 HTTP GET request와 HTTP/1.1 200 OK response가 필터 결과로 나타난다.

HTTP request 분석

HTTP request에는 메소드, path, HTTP version, Host header, User-Agent header 등 여러 정보가 들어 있다. URL로 웹사이트에 들어간 요청은 GET 요청이며, 이 경우 강의 예시에서는 별도의 HTTP body가 없다고 설명한다. User-Agent는 접속한 디바이스와 브라우저 환경을 나타내는 header이며, 모바일과 노트북에서 다른 화면을 보여 주는 데도 활용될 수 있다.

request 요소실습에서 확인하는 의미
GET브라우저가 URL로 서버 자원을 요청했다.
/루트 경로에 접근했다.
Host요청 대상 서버 주소가 들어 있다.
User-Agent브라우저와 디바이스 환경 정보가 들어 있다.
Body 없음단순 GET 요청이라 본문이 비어 있다.

HTTP response 분석

서버가 보낸 response는 HTTP version과 status code로 시작한다. 강의 예시에서는 HTTP/1.1 200 OK가 보이며, 이는 요청이 정상적으로 처리되었다는 의미다. 그 아래에는 Date header, 서버 정보 등 여러 header가 있고, 빈 줄 뒤에 response body가 온다.

실습에서는 response body에 HTML 코드가 담겨 오는 것을 확인한다. 브라우저는 이 HTML을 읽어 화면에 보기 좋게 렌더링한다. 멘토는 개발자 도구를 열어 response 본문으로 온 H1 태그가 실제 화면의 HTML과 연결되는 것을 확인한다.

Frame, Ethernet, IPv4, TCP, HTTP

Wireshark에서 하나의 HTTP 패킷을 열면 Frame, Ethernet, IPv4, TCP, HTTP가 계층적으로 보인다. 이것은 앞 강의에서 배운 캡슐화 구조를 그대로 보여 준다. Frame은 데이터 링크 계층의 전체 단위이고, Ethernet은 MAC 주소를 기반으로 근거리 네트워크에서 전달하기 위한 링크 계층 정보를 담는다. IPv4는 네트워크 계층 데이터그램 헤더이고, TCP는 전송 계층 세그먼트 헤더이며, HTTP는 응용 계층 메시지다.

Ethernet과 IPv4 헤더 관찰

Ethernet 헤더는 보통 14바이트로 설명된다. 6바이트 destination MAC, 6바이트 source MAC, 그리고 상위 계층 프로토콜을 나타내는 type 필드가 있다. type이 IPv4로 표시되면 Ethernet 프레임의 payload가 IPv4 데이터그램이라는 뜻이다.

IPv4 헤더에서는 source address와 destination address를 확인한다. 서버에서 클라이언트로 오는 response라면 source는 서버 IP, destination은 클라이언트 IP가 된다. 또 protocol 필드가 TCP를 가리킬 수 있고, TTL은 패킷이 네트워크에서 무한히 돌지 않도록 제한하는 값이다. total length, header length, checksum 같은 필드도 함께 볼 수 있다.

TCP 헤더와 HTTP payload

TCP 헤더에서는 source port와 destination port를 확인한다. 서버에서 클라이언트로 오는 응답이라면 서버의 port가 source port가 되고, 클라이언트 측 수신 port가 destination port가 된다. 강의 예시에서는 서버가 8080 port를 사용하므로 source port 8080을 볼 수 있다고 설명한다.

TCP의 sequence number와 acknowledgement number는 신뢰성 있는 연결과 확인 응답을 이해하는 데 중요하다. TCP payload 부분에는 HTTP 메시지가 들어 있다. 결국 HTTP header와 body는 TCP 세그먼트의 data로 들어가고, TCP 세그먼트는 IP 데이터그램에, IP 데이터그램은 Ethernet 프레임에 들어가는 구조다.

IP 필터와 Follow TCP Stream

특정 서버와 오간 모든 패킷을 보려면 display filter를 사용할 수 있다. 강의에서는 서버 IP를 기준으로 ip.addr == 12.46.96.85 형태의 필터를 설명한다. Wireshark 필터 문법은 ip.addr == 주소 형식으로 이해하면 된다.

HTTP 패킷을 우클릭해 Follow TCP Stream을 선택하면 하나의 TCP 연결에서 request와 response가 어떤 메시지 흐름으로 오갔는지 볼 수 있다. 이 기능은 HTTP 필터로 request/response를 확인하는 것과 함께, 실제 애플리케이션 데이터 흐름을 보는 데 유용하다.

Flow Graph와 3-way handshake

TCP는 데이터를 보내기 전에 3-way handshake로 연결 설정을 한다. 강의는 특정 IP로 필터를 건 뒤 Statistics의 Flow Graph를 열고, limit to display filter를 적용해 해당 통신의 흐름만 그래프로 확인한다. 그러면 HTTP request가 가기 전에 SYN, SYN/ACK, ACK가 먼저 오가는 것을 볼 수 있다.

→

연결 후 데이터 교환

만약 실습 화면에서 3-way handshake가 잘 보이지 않는다면, 이미 연결이 수립되어 있을 수 있다. 강의에서는 브라우저를 껐다가 다시 켜고 캡처를 다시 해 보면 연결 설정부터 볼 수 있다고 안내한다.

실습의 결론

이 실습은 HTTP request와 response, 계층별 캡슐화, TCP 연결 설정을 한 번에 확인하는 것이 목적이다. 패킷 분석을 할 때는 단순히 payload만 보는 것이 아니라, Ethernet, IP, TCP, HTTP가 각각 어떤 헤더와 의미를 갖는지 함께 읽어야 한다. 그래야 프로토콜 분석, 침해사고 대응, 포렌식, 취약점 분석에서 실제 네트워크 증거를 해석할 수 있다.
