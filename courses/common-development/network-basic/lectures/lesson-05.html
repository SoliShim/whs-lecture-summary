SECTION 01

전송 계층의 위치

앞에서 데이터 링크 계층은 근거리 네트워크의 이웃 요소로 프레임을 전달했고, 네트워크 계층은 전송 계층 세그먼트를 데이터그램으로 캡슐화해 호스트 대 호스트 전달과 라우팅을 담당한다고 배웠다. 전송 계층은 그보다 상위에서 응용 프로그램 메시지를 운반한다.

브라우저, 게임, 여러 네트워크 애플리케이션에서 발생하는 메시지는 전송 계층 헤더가 붙어 세그먼트가 된다. 전송 계층은 클라이언트와 서버의 애플리케이션 계층 메시지를 주고받게 하는 서비스를 제공한다. 대표 프로토콜은 TCP와 UDP이며, 현재도 매우 많이 사용된다.

SECTION 02

프로세스 간 논리적 통신

네트워크 계층이 호스트 대 호스트 논리적 통신을 제공한다면, 전송 계층은 프로세스 대 프로세스 논리적 통신을 제공한다. 사용자는 마치 두 애플리케이션이 직접 연결된 것처럼 느끼지만, 실제로는 하위 계층들이 제공하는 서비스에 의존해 통신이 이루어진다.

강의에서는 TCP, UDP 외에도 QUIC을 언급한다. QUIC은 Google Chrome 등에서 사용되는 범용 목적의 전송 계층 프로토콜로, UDP 기반 위에 필요한 기능을 올려 TCP와 UDP의 장단점을 조화롭게 활용하려는 방향으로 설명된다.

SECTION 03

소켓, 포트, 다중화와 역다중화

목적지의 전송 계층은 네트워크 계층으로부터 세그먼트를 수신한다. 그리고 이것을 소켓으로 전달해 애플리케이션이 사용할 수 있게 한다. 소켓은 애플리케이션과 전송 계층 사이의 인터페이스이며, 포트 번호는 소켓을 식별하는 중요한 값이다.

용어의미
소켓애플리케이션과 전송 계층 사이의 통신 출입구다.
포트 번호호스트 안에서 어떤 프로세스 또는 소켓으로 전달할지 구분하는 식별자다.
역다중화들어온 세그먼트를 알맞은 소켓으로 나누어 전달하는 과정이다.
다중화여러 소켓의 데이터를 아래 네트워크 계층으로 모아 내려보내는 과정이다.

전송 계층이 프로세스 대 프로세스 전달을 담당하기 때문에, 다중화와 역다중화는 매우 중요한 개념이다.

SECTION 04

UDP

UDP는 간소화된 전송 계층 프로토콜이다. 기본적으로 비연결형이고 비신뢰적이다. 다만 기능이 전혀 없는 것은 아니다. 다중화·역다중화 기능을 제공하고, 체크섬을 통해 비트 오류를 검출할 수 있다.

UDP의 장점은 빠르고 효율적이라는 점이다. TCP와 달리 handshake 연결 설정 과정이 없고, 연결 상태를 유지하지 않으며, 헤더와 처리 구조도 작다. 또한 애플리케이션 계층에서 필요한 기능만 직접 구현해 정교하게 제어할 수 있다. 그래서 스트리밍처럼 실시간성이 중요한 애플리케이션에서 유리할 수 있다.

하지만 UDP에는 혼잡 제어가 없다. 네트워크가 감당할 수 없는 양의 패킷이 쏟아지면 큰 문제가 될 수 있는데, UDP 자체는 이를 조절하는 메커니즘을 제공하지 않는다. 따라서 UDP의 단순함은 장점이자 동시에 조심해야 할 단점이다.

SECTION 05

TCP

TCP는 UDP와 달리 신뢰성 있는 데이터 전송과 연결 지향형 서비스를 제공한다. 클라이언트가 서버에 데이터를 보내면 서버의 확인 응답을 받고, 이를 바탕으로 다음 데이터를 보내거나 손실을 판단한다. 이렇게 양방향에서 확인 응답을 주고받기 때문에 신뢰성 있는 전송이 가능하다.

TCP는 데이터를 보내기 전에 TCP 3-way handshake로 연결을 설정한다. 연결이 수립된 상태에서 데이터를 주고받는다. 또 흐름 제어와 혼잡 제어 기능을 제공한다. 흐름 제어는 송신자와 수신자 사이에서 수신자가 감당하기 어려운 과부하가 생기지 않도록 조절하는 것이고, 혼잡 제어는 네트워크 전체가 감당할 수 없을 정도로 패킷이 많아지는 것을 방지하는 메커니즘이다.

필드·개념역할
source port보낸 프로세스의 소켓 식별에 쓰인다.
destination port받는 프로세스의 소켓 식별에 쓰인다.
sequence number세그먼트에 담긴 첫 번째 바이트의 스트림 번호다.
acknowledgement number상대에게 다음에 받고 싶은 바이트 번호를 알려 주는 기대값이다.
data응용 계층 메시지가 들어간다.
SECTION 06

MSS, MTU, segmentation

전송해야 할 데이터가 5만 바이트처럼 크다면 한 세그먼트에 모두 담을 수 없다. TCP는 세그먼트에 담을 수 있는 최대 데이터 양을 기준으로 데이터를 쪼갠다. 이 값을 MSS, Maximum Segment Size라고 한다.

강의에서는 MSS가 MTU를 기준으로 결정된다고 설명한다. MTU는 링크 계층에서 프레임 전체가 가질 수 있는 최대 크기이며, MSS는 MTU에서 IP 헤더와 TCP 헤더를 뺀 크기라고 이해하면 된다. 데이터가 MSS를 넘으면 TCP는 segmentation을 수행해 여러 세그먼트로 나누어 보낸다.

SECTION 07

Sequence와 acknowledgement

Sequence number는 지금 보내는 데이터가 바이트 스트림에서 어디부터 시작하는지 나타낸다. 예를 들어 전체 데이터 스트림이 50만 바이트이고 MSS가 1000바이트라면, TCP는 약 500개의 세그먼트로 나누어 보낼 수 있다. 각 세그먼트는 자기 첫 번째 바이트 위치를 sequence number로 가진다.

A가 B에게 sequence number 42와 acknowledgement number 79를 보냈다면, A가 보내는 세그먼트는 42번째 바이트부터 시작하고, A는 B에게서 79번 바이트부터의 데이터를 기대한다는 뜻으로 이해할 수 있다. B가 42번 바이트를 받았다면 다음에 받고 싶은 값인 43을 acknowledgement number로 보낼 수 있다. 이렇게 sequence와 acknowledgement가 맞물리며 순서와 중복 없는 전송을 돕는다.

SECTION 08

신뢰성, timeout, RTT

TCP의 신뢰성은 중복이 없고 순서가 유지되는 데이터 전송을 보장하는 방향으로 구현된다. 손실된 세그먼트를 발견하기 위한 중요한 메커니즘 중 하나가 timeout이다. 송신자가 서버로 값을 보냈는데 일정 시간 동안 확인 응답이 오지 않으면 패킷 손실이 일어났다고 판단하고 재전송할 수 있다.

RTT는 Round Trip Time, 왕복 시간이다. 내가 보낸 데이터가 상대에게 갔다가 응답이 돌아오는 데 걸리는 시간이다. timeout은 당연히 RTT보다 커야 한다. timeout이 RTT보다 너무 작으면 정상 응답도 모두 손실로 오해할 수 있기 때문이다.

SECTION 09

TCP 3-way handshake와 4-way termination

TCP는 데이터를 보내기 전에 3-way handshake로 연결을 설정한다. 먼저 클라이언트가 SYN 플래그를 켜서 서버에게 연결 요청을 보낸다. 서버는 SYN과 ACK를 함께 보내 연결 요청을 확인하고 자기 쪽도 연결을 열겠다고 응답한다. 마지막으로 클라이언트가 ACK를 보내면 연결이 수립되고 데이터 전송이 가능해진다.

클라이언트가 연결 요청

→
2. SYN/ACK

서버가 요청 확인과 연결 동의

→
3. ACK

클라이언트가 연결 완료

연결 종료는 보통 4-way handshake로 이루어진다. 한쪽이 FIN 플래그로 종료 의사를 보내면 상대가 ACK로 확인하고, 상대도 FIN을 보내 종료 의사를 밝히며, 마지막 ACK로 종료가 마무리된다. 종료는 클라이언트와 서버 어느 쪽에서도 먼저 시작할 수 있다.

SECTION 10

TCP와 UDP를 구분해야 하는 이유

강의는 TCP와 UDP의 차이를 정확히 아는 것이 네트워크 프로그래밍과 보안 취약점 분석에서 매우 중요하다고 강조한다. TCP는 신뢰성, 연결 지향, 흐름 제어, 혼잡 제어를 제공하지만 구조가 복잡하다. UDP는 단순하고 빠르지만 신뢰성이나 혼잡 제어가 없다. 어떤 프로토콜을 쓰느냐에 따라 패킷 분석 방법, 장애 원인, 공격 가능성이 달라진다.

챕터 5 트랜스포트 계층을 시작하도록 하겠습니다. 저희가 지금 네트워크 계층 구조에 대해서 TCP IP 계층 구조에 대해서 지금 이해를 했고 데이터 링크 레이어에 대해서 어떤 역할을 하는지 그리고 네트워크 레이어에서 또 어떤 역할을 하는지 이해왔습니다. 데이터 링크 레이어는 프레임이라고 하는 한 요소 네트워크에서 이윤 네트워크 요소로 프레임을 전달하는 근거래 네트워크에서 프레임을 전달하는 내용을 이해를 해봤고요. 네트워크 레이어는 호스트 대 호스트로 세그먼트 이 트랜스포트 계층의 세그먼트라고 하는 단위를 데이터그램으로 캡슐화해서 어떻게 라우팅하고 이것을 전달할 것인가에 대한 내용을 가볍게 다뤄봤습니다. 이제 저희가 다룰 트랜스포트 계층은 전송 계층인데요. 이것은 저희가 사용하는 응용 프로그램 브라우저나 게임이나 이런 어플리케이션에서 발생하는 메시지를 어떻게 운반하는지에 대한 내용을 다루게 될 것입니다. 좀 더 하이레벨로 이제 온 것이죠. 그래서 이 트랜스포트 레이어는 세그먼트라고 하는 단위를 사용해서 응용 프로그램 계층의 메시지를 전달하게 됩니다.

세그먼트는 이 메시지라는 응용 프로그램 계층의 메시지를 트랜스포트 레이어 헤더를 붙여서 구성된 것이라고 할 수 있고요. 단위는 세그먼트입니다. 클라이언트와 서버 간의 어플리케이션 계층의 메시지 저희가 웹사이트를 들어가거나 할 때 발생하는 그 메시지를 전송하는 서비스를 제공하게 됩니다. 그래서 저희가 주로 다룰 프로토콜은 TCP와 UDP가 있고요. 현재도 굉장히 많이 쓰이고 있는 프로토콜입니다. 간단히 소개를 하자면 UDP가 조금 가벼운 버전이고요. TCP는 뭔가 더 서비스가 많은 프로토콜이라고 할 수 있습니다. 대표적으로 이 TCP는 저희가 어플리케이션을 사용하면서 뭔가 신뢰할 수 있게 연결 지향형 서비스를 제공하고요. 그리고 흐름 제어나 혼합 제어 같이 어떤 속도를 일치시키거나 전송료를 컨트롤하는 그런 기능들이 추가적으로 되어 있습니다. 하지만 UDP는 그런 기능을 제공하진 않지만 그리고 비연결형 서비스를 제공하지만 되게 빠르고 효율적으로 사용할 수 있게 하는 그런 프로토콜이라고 할 수 있습니다. 그 앞서 배웠던 네트워크 레이어는 호스트 대 호스트 논리적 통신을 제공한다고 제가 설명을 드렸었는데요.
