SECTION 01

전송 계층의 위치

5강은 앞 강의의 계층 복습에서 시작한다. 데이터 링크 계층은 하나의 로컬 네트워크 안에서 frame을 이웃 네트워크 요소로 옮기는 역할을 했고, 네트워크 계층은 transport segment를 datagram으로 캡슐화해 한 host에서 다른 host까지 전달하고 route를 선택하는 역할을 했다. 전송 계층은 이 네트워크 계층 위에서 동작하며, 실제 응용 프로그램이 만든 메시지를 프로세스 단위로 전달하는 계층이다.

슬라이드의 단위는 segment다. 브라우저, 게임, 메신저, 서버 프로그램 같은 application process가 만든 message가 전송 계층으로 내려오면, 여기에 transport-layer header가 붙어 segment가 된다. 이 segment가 아래 네트워크 계층으로 내려가 datagram에 담기고, 다시 링크 계층 frame에 담겨 실제 네트워크를 지나간다.

강사는 전송 계층의 대표 프로토콜로 TCP와 UDP를 제시한다. TCP는 연결 지향형이고 신뢰성 있는 전송, 흐름 제어, 혼잡 제어, segmentation을 제공한다. UDP는 더 가볍고 연결 설정이 없으며 빠르지만, 신뢰성·흐름 제어·혼잡 제어를 기본으로 제공하지 않는다. 이 차이는 이후 패킷 분석과 네트워크 프로그래밍에서 계속 기준점이 된다.

SECTION 02

프로세스 간 논리적 통신

전송 계층을 이해할 때 가장 중요한 표현은 process-to-process logical communication이다. 네트워크 계층은 host-to-host logical communication을 제공한다. 즉, 호스트 A에서 호스트 B까지 패킷이 갈 수 있게 해 준다. 하지만 한 호스트 안에는 브라우저, 게임 클라이언트, 백그라운드 서비스, 서버 데몬처럼 여러 프로세스가 동시에 실행된다. 전송 계층은 그중 어떤 프로세스와 어떤 프로세스가 통신하는지를 구분해 준다.

슬라이드에서는 하위 계층이 제공하는 서비스를 이용하지만, 애플리케이션 입장에서는 서로 다른 host의 process가 직접 연결된 것처럼 보인다고 설명한다. 그래서 전송 계층의 통신은 실제 물리적 직접 연결이 아니라 논리적 연결이다. 중간에는 라우터, 링크, 네트워크 계층, 데이터 링크 계층이 있지만, 개발자와 사용자는 대개 process 사이의 데이터 흐름으로 바라본다.

강의는 TCP와 UDP 외에 QUIC도 함께 언급한다. QUIC은 Google Chrome 등에서 활용되는 범용 목적의 전송 계층 프로토콜로 소개되며, UDP 기반 위에 신뢰성·연결 관리 같은 기능을 올려 TCP와 UDP의 장점을 조합하려는 방향으로 이해하면 된다. 이 강의의 중심은 TCP와 UDP지만, QUIC 언급은 현대 웹 트래픽이 전통적인 TCP만으로 설명되지 않을 수 있음을 보여 준다.

SECTION 03

소켓, 포트, 다중화와 역다중화

목적지 호스트에 segment가 도착하면, 전송 계층은 네트워크 계층으로부터 그 segment를 받아 올바른 socket으로 전달한다. socket은 application process와 전송 계층 사이에 있는 통신 출입구다. 강사는 socket을 애플리케이션과 transport/network layer 사이의 중간 인터페이스처럼 설명한다.

여기서 port number가 등장한다. 한 host 안에는 여러 socket이 있을 수 있고, 전송 계층은 port number를 이용해 어느 socket으로 데이터를 전달해야 하는지 식별한다. 예를 들어 웹 서버, SSH 서버, DNS 서버가 같은 호스트에서 동시에 동작할 수 있는 이유는 전송 계층이 포트 번호로 프로세스의 통신 출입구를 구분하기 때문이다.

용어의미
소켓애플리케이션과 전송 계층 사이의 통신 출입구다.
포트 번호호스트 안에서 어떤 프로세스 또는 소켓으로 전달할지 구분하는 식별자다.
역다중화, demultiplexing들어온 segment를 port number 등을 기준으로 알맞은 socket에 나누어 전달하는 과정이다.
다중화, multiplexing여러 socket에서 나온 데이터를 전송 계층이 모아 segment로 만들고 아래 네트워크 계층으로 내려보내는 과정이다.

강의의 그림은 receiving host에서 transport layer가 segment를 받아 socket으로 넘기는 모습을 보여 준다. 이 흐름이 역다중화다. 반대로 송신 측에서는 여러 process가 만든 데이터를 transport layer가 받아 하나의 네트워크 계층 인터페이스로 내려보내므로 다중화가 일어난다. 전송 계층이 process-to-process 전달을 맡는다는 말은 결국 이 socket과 port number 처리까지 포함한다.

SECTION 04

UDP

UDP, User Datagram Protocol은 간소화된 전송 계층 프로토콜이다. 강의에서는 UDP를 unreliable, connectionless protocol이라고 정리한다. 즉, TCP처럼 연결을 먼저 맺지 않고, 데이터를 보낸 뒤 수신 확인·재전송·순서 보장을 전송 계층이 책임지지 않는다.

그렇다고 UDP가 아무 역할도 하지 않는 것은 아니다. UDP는 전송 계층의 기본 기능인 multiplexing과 demultiplexing을 제공한다. 또한 checksum 필드를 통해 segment 안의 bit error를 검출할 수 있다. 슬라이드의 UDP header는 source port, dest port, length, checksum 네 필드로 구성되어 있고, 각 필드는 16bit다. 그래서 UDP header는 총 8byte로 매우 작다.

UDP 필드역할
source port송신 측 socket 또는 process를 식별하는 데 사용된다.
dest port수신 측 host 안에서 어느 socket으로 보낼지 구분한다.
lengthUDP header와 data를 포함한 UDP segment의 길이를 나타낸다.
checksum전송 중 bit error가 생겼는지 검출하는 데 사용된다.

UDP는 실시간성이 중요한 애플리케이션에서 자주 언급된다. 스트리밍, 음성·영상 통화, 게임처럼 조금의 손실보다 지연이 더 치명적인 경우에는 TCP식 재전송과 순서 보장이 오히려 불리할 수 있다. 이런 경우 애플리케이션이 필요한 보정만 직접 구현하고, 전송 계층은 가볍게 가져가는 설계가 가능하다.

슬라이드의 장점은 네 가지로 정리된다. 첫째, application layer에서 보낼 데이터와 보낼 시점을 더 세밀하게 제어할 수 있다. 둘째, connection establishment가 없어 handshake 지연이 없다. 셋째, connection state를 유지하지 않으므로 서버가 많은 클라이언트를 상대할 때 부담이 작다. 넷째, TCP header가 보통 20byte인 것과 비교해 UDP header는 8byte라 header overhead가 작다.

단점도 분명하다. 강의는 congestion control이 없다는 점을 강조한다. 네트워크가 감당할 수 없을 정도로 패킷이 쏟아질 때 UDP 자체는 속도를 줄이거나 네트워크 혼잡을 완화하는 기능을 제공하지 않는다. 그래서 UDP의 빠름과 단순함은 장점이지만, 보안·장애 분석 관점에서는 트래픽 폭주와 서비스 품질 문제를 함께 떠올려야 한다.

SECTION 05

TCP

TCP, Transmission Control Protocol은 UDP와 반대로 신뢰성 있는 데이터 전송과 연결 지향형 서비스를 제공한다. 강사는 TCP가 수신 여부 확인, 재전송, packet 순서 유지에 sequence number와 acknowledgement number를 사용한다고 설명한다. 데이터를 그냥 던지는 것이 아니라, 상대가 받았는지 확인하고 필요하면 다시 보내며, byte stream의 순서를 유지한다는 점이 핵심이다.

TCP는 데이터를 보내기 전에 handshake로 연결을 설정한다. 연결이 수립된 상태에서 양쪽 process가 데이터를 주고받는다. 또한 TCP는 flow control과 congestion control을 제공한다. flow control은 송신자와 수신자 사이의 packet 수를 조절해 수신 측이 감당하지 못하는 상황을 줄이는 기능이고, congestion control은 네트워크 내부 packet 수를 조절해 네트워크 overflow를 방지하는 기능이다.

필드·개념역할
source port보낸 프로세스의 소켓 식별에 쓰인다.
destination port받는 프로세스의 소켓 식별에 쓰인다.
sequence number세그먼트에 담긴 첫 번째 바이트의 스트림 번호다.
acknowledgement number상대에게 다음에 받고 싶은 바이트 번호를 알려 주는 기대값이다.
receive window수신 측이 받을 수 있는 양을 알려 flow control에 사용된다.
flagsSYN, ACK, FIN 같은 제어 bit로 연결 시작·응답·종료 흐름을 표현한다.

슬라이드의 TCP header 그림은 source port와 dest port 아래에 sequence number, acknowledgement number가 크게 놓여 있음을 보여 준다. 이 두 필드는 뒤에서 배우는 신뢰성 있는 전송의 중심이다. header length, receive window, internet checksum, urgent data pointer, options, data 영역도 함께 표시되어 TCP가 UDP보다 훨씬 많은 상태와 제어 정보를 다룬다는 점을 확인할 수 있다.

SECTION 06

MSS, MTU, segmentation

전송해야 할 application data가 크면 한 TCP segment에 모두 담을 수 없다. TCP는 데이터를 여러 segment로 나누는데, 이때 한 segment의 payload 영역에 담을 수 있는 최대 application data 크기를 MSS, Maximum Segment Size라고 한다. 슬라이드의 표기처럼 MSS는 header를 포함한 TCP segment 전체 크기가 아니라, application data에 대한 최대 크기다.

MSS를 결정하는 중요한 요소가 MTU, Maximum Transmission Unit이다. 강의에서는 MTU를 로컬 송신 host가 전송할 수 있는 가장 큰 링크 계층 frame 길이로 설명한다. 네트워크 경로 중 MTU가 서로 다른 구간이 있으면, 그 경로에서 사용할 수 있는 더 작은 MTU를 기준으로 MSS를 조정하기도 한다.

기본 계산은 MSS = MTU - TCP/IP header다. 슬라이드 예시는 MTU가 1500byte이고 TCP/IP header가 40byte이면 MSS가 1460byte가 된다고 보여 준다. 이 40byte는 보통 IPv4 header 20byte와 TCP header 20byte를 합친 값으로 이해하면 된다. 따라서 TCP가 50,000byte 같은 큰 데이터를 보내야 한다면, MSS 단위로 잘라 여러 segment를 만든 뒤 sequence number로 각 조각의 위치를 관리한다.

SECTION 07

Sequence와 acknowledgement

TCP는 byte stream을 기준으로 데이터를 관리한다. Sequence number는 어떤 segment에 들어 있는 첫 번째 byte가 전체 byte stream에서 몇 번째 위치인지 나타낸다. 예를 들어 데이터 스트림이 500,000byte이고 MSS를 1,000byte로 가정하면, segment는 총 500개로 구성된다. 첫 번째 segment는 0번 byte부터, 두 번째 segment는 1,000번 byte부터 시작하는 식으로 번호가 붙는다.

Acknowledgement number는 상대방에게서 기대하는 다음 byte의 sequence number다. 슬라이드 오른쪽 예시에서 Host A가 Host B에게 Seq=42, ACK=79, data='C'를 보낸다. 이것은 A가 보내는 데이터가 byte stream의 42번 위치에서 시작하며, 동시에 A는 B에게서 다음으로 79번 byte를 기대하고 있다는 뜻이다.

B가 A의 'C'를 받으면, B는 A에게 Seq=79, ACK=43, data='C'를 보낸다. 여기서 ACK=43은 A가 보낸 42번 byte를 받았으므로 다음에는 43번 byte를 기대한다는 의미다. 마지막으로 A는 B의 응답을 받고 Seq=43, ACK=80 흐름으로 이어 간다. 강의의 핵심은 ACK가 “방금 받은 번호”가 아니라 다음에 받고 싶은 번호라는 점이다.

이 구조 덕분에 TCP는 순서를 유지하고, 누락된 부분을 찾고, 같은 데이터가 중복되어 들어왔는지 판단할 수 있다. 따라서 sequence number와 acknowledgement number는 TCP의 신뢰성을 설명할 때 반드시 함께 이해해야 한다.

SECTION 08

신뢰성, timeout, RTT

TCP가 제공하는 신뢰성 있는 데이터 전송은 “중복이 없고 순서가 유지되는 것”을 보장하는 방향으로 설명된다. 이를 위해 TCP는 ACK, sequence number, 재전송을 함께 사용한다. segment를 보냈는데 일정 시간 안에 ACK를 받지 못하면, TCP는 해당 segment가 손실되었을 가능성이 있다고 판단하고 다시 전송할 수 있다.

슬라이드에서는 이 메커니즘을 retransmission timeout이라고 부른다. timeout 값은 연결의 왕복 시간인 RTT, Round Trip Time보다 조금 더 커야 한다. RTT는 송신 측에서 segment를 전송한 시점부터 그 segment에 대한 긍정 확인 응답, 즉 ACK를 받을 때까지 걸리는 시간이다.

timeout이 RTT보다 작으면 정상적으로 돌아오는 ACK도 너무 늦었다고 오해해 불필요한 재전송이 발생한다. 반대로 timeout이 지나치게 크면 실제 손실이 발생했을 때 복구가 늦어진다. 강의에서는 TCP가 손실 segment를 발견하기 위해 timeout/재전송 메커니즘을 사용한다는 점과, timeout을 RTT보다 조금 크게 잡아야 한다는 점을 핵심으로 잡으면 된다.

SECTION 09

TCP 3-way handshake와 4-way termination

TCP는 connection-oriented protocol이기 때문에 데이터를 본격적으로 보내기 전에 연결을 먼저 설정한다. 이 연결 시작 과정이 3-way handshake다. 강의에서는 client host와 server host 사이의 세 메시지를 순서대로 설명한다.

  1. Client → Server: 연결 요청을 보낸다. SYN bit가 1이고, client initial sequence number를 담아 syn=1, seq=a 형태가 된다.
  2. Server → Client: 서버가 응답한다. 서버도 SYN bit를 1로 켜고 자기 sequence number를 담으며, 클라이언트의 요청을 확인하기 위해 ack=a+1을 보낸다. 즉 syn=1, seq=b, ack=a+1이다.
  3. Client → Server: 클라이언트가 연결 완료를 알린다. 이때 SYN bit는 0으로 내려가고, seq=a+1, ack=b+1 흐름이 된다. 강의에서는 세 번째 메시지에는 payload가 포함될 수 있다고 덧붙인다.

연결을 닫을 때는 4-way termination, 또는 4-way handshake로 종료한다. 시작과 달리 종료는 두 프로세스 중 어느 쪽이든 먼저 시작할 수 있다. 한쪽이 더 이상 보낼 데이터가 없다는 뜻으로 FIN을 보내고, 상대가 ACK로 확인한 뒤, 상대도 FIN을 보내고 마지막 ACK를 받으면 연결이 닫힌다.

  1. Client → Server: 연결 종료 요청을 보낸다. FIN bit가 1이다.
  2. Server → Client: 서버가 요청을 확인하며 ACK를 보낸다.
  3. Server → Client: 서버도 연결 종료 요청을 보낸다. FIN bit가 1이다.
  4. Client → Server: 클라이언트가 마지막 ACK를 보내 종료를 확인한다.

슬라이드는 종료 뒤 timed wait과 closed 상태를 함께 보여 준다. 강사는 연결이 끝나면 host의 자원, 예를 들어 buffer와 memory가 회수된다고 설명한다. 따라서 TCP 연결은 단순히 패킷 몇 개를 주고받는 절차가 아니라, 양쪽 host가 연결 상태와 자원을 관리하는 과정이다.

세그먼트는 이 메시지라는 응용 프로그램 계층의 메시지를 트랜스포트 레이어 헤더를 붙여서 구성된 것이라고 할 수 있고요. 단위는 세그먼트입니다. 클라이언트와 서버 간의 어플리케이션 계층의 메시지 저희가 웹사이트를 들어가거나 할 때 발생하는 그 메시지를 전송하는 서비스를 제공하게 됩니다. 그래서 저희가 주로 다룰 프로토콜은 TCP와 UDP가 있고요. 현재도 굉장히 많이 쓰이고 있는 프로토콜입니다. 간단히 소개를 하자면 UDP가 조금 가벼운 버전이고요. TCP는 뭔가 더 서비스가 많은 프로토콜이라고 할 수 있습니다. 대표적으로 이 TCP는 저희가 어플리케이션을 사용하면서 뭔가 신뢰할 수 있게 연결 지향형 서비스를 제공하고요. 그리고 흐름 제어나 혼합 제어 같이 어떤 속도를 일치시키거나 전송료를 컨트롤하는 그런 기능들이 추가적으로 되어 있습니다. 하지만 UDP는 그런 기능을 제공하진 않지만 그리고 비연결형 서비스를 제공하지만 되게 빠르고 효율적으로 사용할 수 있게 하는 그런 프로토콜이라고 할 수 있습니다. 그 앞서 배웠던 네트워크 레이어는 호스트 대 호스트 논리적 통신을 제공한다고 제가 설명을 드렸었는데요.

트랜스포트는 좀 더 고차원적으로 프로세스 간의 논리적 통신을 제공합니다. 그러니까 어떤 프로그램에서 발생하는 그런 메시지들을 이 계층만 봤을 때는 프로세스와 프로세스가 논리적 통신이 되게끔 하는 계층이고요. 이게 가능하기 위해서는 앞서 배웠던 그런 계층들이 계층들을 의존해서 트랜스포트 계층은 수행을 하게 됩니다. 그래서 사실 직관적으로 이 프로세스들이 통신을 할 때 상위 레벨에서 보면 그냥 직접적으로 연결된 것처럼 느껴지는 것이죠. 그래서 대표적인 트랜스포트 계층의 프로토콜은 TCP, UDP, QUIC이 있습니다. TCP, UDP는 제가 앞서 간단히 설명을 드렸는데 QUIC이라고 하는 것은 구글 크롬에서 사용하는 범연 목적의 트랜스포트 계층 프로토콜입니다. 아무래도 TCP의 장단점, UDP의 장단점이 있다 보니까 이것들을 좀 조화롭게 해가지고 UDP 기반의 QUIC이라는 프로토콜을 만들어서 필요한 서비스들만 추가해서 사용하는 그런 형태로 발전한 프로토콜이라고 이해하시면 될 것 같습니다. 다중화, 역다중화를 다루기 앞서서 이제 하나 설명을 드릴게요.

지금 이 사진을 한번 봅시다 먼저 목적지의 트랜스포트 계층은 아래 네트워크 계층으로부터 세그먼트를 수신합니다. 지금 두 번째 컴퓨터를 보시면 이 트랜스포트 레이어는 네트워크 레이어로부터 세그먼트를 수신해 오겠죠. 그러면 그것을 소켓이라고 하는 이 어플리케이션과 네트워크 레이어의 그 중간에 있는 중간 매개자인 소켓으로 전달을 해서 이 어플리케이션에 전달을 하게 됩니다. 그래서 이 소켓은 포트 넘버라는 것으로 이루어져 있고요. 유일한 소켓의 식별자라고 이해하시면 됩니다. 그래서 다중화와 역다중화가 무엇이냐라고 하면 어떤 물리 매체를 이용해서 다양한 패킷들이 오갈 텐데 그거를 알맞게 알맞은 소켓에다가 전달하는 것을 퍼지는 느낌이죠. 그거를 역다중화라고 하고요. 그리고 반대로 내가 어떤 값을 전송할 때 그 아래 계층 네트워크 계층으로 전달할 때는 다중화를 해서 이렇게 모아서 보내는 것을 다중화라고 할 수 있습니다. 트랜스포트 계층은 프로세스 대 프로세스 전달을 하는 계층이기 때문에 이러한 다중화와 역다중화에 대한 개념을 이해할 필요가 있습니다.

UDP에 대해서 설명드리도록 하겠습니다. UDP는 아까 말씀드렸듯이 조금 간소화된 버전의 프로토콜이라고 할 수 있는데요. UDP는 기본적으로 비실례적, 비연결적인 프로토콜입니다. 간단한 구조, 그냥 값을 보내는 정도의 프로토콜이라고 이해를 하시면 되는데 그렇다고 UDP가 아예 기능이 없는 것은 아닙니다. 아까 말씀드린 다중화 역다중화 기능은 기본적으로 가지고 있고 이 아래 사진을 보시면 체크섬이라고 하는 필터가 있어서 비트가 만약에 오류가 발생했을 때 검출할 수 있는 그런 서비스를 제공합니다. 그래서 이렇게 UDP같이 간단한 프로토콜은 스트리밍 사이트같이 실시간 어플리케이션에서 굉장히 유리한 장점을 가지고 있게 되는데요. 이 UDP의 장점은 어플리케이션 계층에서 정교하게 제어가 가능하다는 겁니다. 보통은 어플리케이션 계층에서 이런 프로토콜을 직접 컨트롤하기가 쉽지 않지만 UDP를 사용하면 불필요한 서비스들은 사용하지 않고 필요한 어떤 추가 기능을 직접 구현을 해서 정교하게 제어가 가능하다는 것이죠. 그리고 UDP는 뒤에서 설명할 TCP와 다르게 핸드쉐이크라는 연결 설정 과정이 없기 때문에 간단하고 불필요한 과정을 줄일 수 있게 되는 거죠.

그리고 연결 상태를 유지하지 않고 패킷 자체의 크기도 적기 때문에 이제 좀 더 빠르고 효율적으로 사용할 수 있는 프로토콜입니다. 하지만 단점도 존재하는데요. 이 UDP 같은 경우는 혼잡 제어라는 기능을 제공하지 않습니다. 근데 이 혼잡 제어는 네트워크에서 굉장히 중요한 고려사항 중 하나로 이 네트워크가 필요한 작업을 할 수 없게 되는 상태에 빠지면 굉장히 큰 문제가 되겠죠. 그래서 UDP는 이러한 장점과 혼잡 제어의 부재라는 단점을 가지고 있음을 여러분들이 이해하셨으면 좋겠습니다. 다음은 TCP 프로토콜에 대해서 설명을 드리도록 하겠습니다. TCP는 비연결적, 비실례적인 UDP와 다르게 신뢰성 있는 데이터를 전송하고요. 그리고 연결 지향형 프로토콜입니다. 이 TCP는 제가 만약에 서버로 어떤 값을 보낸다 했을 때 그 서버로부터 확인응답을 받게 됩니다. 그러면 이 서버가 잘 받았구나라는 것을 인지하고 나는 다음 패킷을 보내야 되고 서버는 얘가 잘 받았네 하면서 양방향에서 신뢰성 있는 데이터 전송이 가능하다는 것이죠. 그리고 연결 지향형입니다.
