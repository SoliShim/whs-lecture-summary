SECTION 01

오류 메시지 노출

프로그램 개발 과정에서는 오류가 발생한다. 문제는 그 오류 메시지가 서버 관리자만 보는 것이 아니라 일반 사용자에게 노출될 때다. SQL 오류 메시지나 Django 기본 오류 페이지에는 어떤 SQL문이 실패했는지, 어떤 경로와 코드가 문제인지, admin 같은 내부 경로가 무엇인지 드러날 수 있다.

공격자는 이런 정보를 힌트로 삼아 다음 공격을 준비한다. 따라서 400, 403, 500 같은 오류 응답이 발생했을 때는 개발자가 직접 설정한 오류 페이지로 연결해야 한다. 사용자에게는 필요한 최소 정보만 보여 주고, 자세한 내부 정보는 서버 로그에만 남기는 방식이 적절하다.

def handler500(request):
    return render(request, "errors/500.html", status=500)

def download(request):
    filename = request.GET.get("filename")
    if not filename or not filename.strip():
        return HttpResponseBadRequest("파일 이름이 필요합니다.")
    return safe_download(filename)
SECTION 02

부적절한 예외 처리

예외가 발생해도 프로그램이 안전하게 동작하도록 설계해야 한다. 예외 처리가 없으면 프로그램이 종료되거나 서버가 중단될 수 있다. 예외 처리를 했더라도 오류 데이터나 경로를 그대로 보여 주면 공격자에게 힌트가 된다.

강의에서는 암호화 과정의 예외 처리를 예로 든다. 키 아이디가 인덱스를 넘어 오류가 발생했는데 단순히 pass로 넘어가면, 코드가 기본 키로 암호화를 수행할 수 있다. 기본 키가 노출되거나 쉽게 추측되면 평문 데이터가 노출될 수 있다. 예외 상황에서는 취약한 기본 동작으로 빠지지 말고, 안전한 새 키를 만들거나 요청을 실패 처리해야 한다.

SECTION 03

널 참조와 코드 오류

코드 작성 과정의 논리 오류도 취약점이 된다. 공격자나 일반 사용자가 객체 값을 비워 둘 수 있는데, 서버가 항상 값이 있다고 가정하면 널 값 처리에서 오류가 발생한다. Python에서는 None 값을 문자열처럼 다루거나, 파일 이름이 없는데 count() 같은 메서드를 호출하면 에러가 발생할 수 있다.

방어하려면 값이 존재하는지, 공백만 들어온 것은 아닌지, 확장자가 의도한 것인지 먼저 확인해야 한다. 값이 없으면 “파일 이름이 필요합니다”처럼 안전한 오류 응답으로 처리하고, 이후 로직으로 넘어가지 않도록 해야 한다.

SECTION 04

역직렬화와 무결성 검증

직렬화된 데이터는 사람이 바로 읽기 어려운 형태로 저장되며, Python의 pickle 파일처럼 로드 과정에서 코드 실행과 비슷한 효과가 날 수 있다. 공격자가 보낸 pickle 파일에 공격 코드가 들어 있는데 무결성 검증 없이 바로 로드하면 서버에서 악성 행위가 실행될 수 있다.

강의는 신뢰할 수 없는 직렬화 데이터는 바로 역직렬화하지 말고, 공식적으로 제공된 해시값과 비교하거나, 데이터가 변조되지 않았는지 먼저 확인해야 한다고 설명한다. 해시값이 다르면 변조된 데이터로 보고 실행하지 않아야 한다.

import hashlib
import pickle

expected_hash = "공식적으로 제공된 SHA-256 해시값"
data = request.body
actual_hash = hashlib.sha256(data).hexdigest()

if actual_hash != expected_hash:
    return HttpResponseBadRequest("변조된 데이터입니다.")

obj = pickle.loads(data)
SECTION 05

DNS Lookup 오용

API 오용의 대표 예로 DNS lookup이 소개된다. 사용자가 www.bank.kr 같은 URL을 입력하면 정상적으로는 DNS 서버가 해당 URL에 맞는 IP를 돌려준다. 하지만 중간에서 공격자가 개입해 다른 IP를 알려 주면, 사용자는 원래 은행 서버가 아니라 공격자의 서버에 접속할 수 있다.

고정된 대상에 접속해야 하는 기능이라면 URL 기반으로만 처리하지 말고, 필요한 경우 신뢰할 수 있는 고정 IP나 인증서 검증 등 더 안전한 확인 절차를 사용해야 한다. 강의에서는 URL보다 IP 주소를 직접 사용하는 방식이 변조 여지를 줄일 수 있다고 설명한다.

SECTION 06

취약한 API와 라이브러리 사용

웹서버는 여러 API와 라이브러리를 사용한다. 사용하는 API 자체나 그 내부 패키지에 취약점이 있다면, API를 호출하는 것만으로도 보안 위험에 노출될 수 있다. 그래서 새로운 API, 패키지, 라이브러리를 사용하기 전에 알려진 취약점이 있는지 확인해야 한다.

가능하면 최신 버전을 사용해야 한다. 업데이트가 계속되는 라이브러리는 취약점이 발견되면 패치가 나오므로, 최신 버전을 유지하는 것 자체가 중요한 시큐어 코딩 습관이다.

시큐어 코딩 7강 시작하겠습니다. 7강에서는 시큐어 코딩 기법에 대해서 계속해서 마지막으로 이야기를 할 것입니다. 목차가 전체 시큐어 코딩 목차가 다음과 같았는데요. 이번 강의에서는 마지막으로 남은 에러 처리, 코드 오류, API 오용에 대한 내용을 다룰 것입니다. 먼저 에러 처리입니다. 프로그램을 개발하는 과정에서 에러가 많이들 발생을 하고 또 에러가 발생을 하면은 그에 해당하는 어디서 에러가 났는지 등을 포함하는 여러가지 오류 메시지들이 존재할 것인데요. 여기 화면에 보이는 것 같은 경우에는 SQL에서 에러가 났을 때 어떤 식으로 에러가 났고 그리고 에러가 나는 SQL문은 무엇이다. 이런 내용의 에러 메시지가 존재할 수 있습니다. 근데 그 에러 메시지가 노출, 에러 메시지가 만약에 서버의 관리자만 볼 수 있는 것이 아니라 일반 사용자들에게도 그것이 노출이 되게 된다면은 이 오류 메시지에서 얻을 수 있는 여러가지 정보들이 많습니다. 그래서 이것이 공격자에게 힌트가 될 수 있는 거죠. 예를 들면은 이제 이거는 장고에서 기본 장고의 기본 오류 메시지를 보여주는 기본 페이지인데요.

딱히 이제 장고에서는 어떤 오류 에러가 발생했을 때 에러 페이지를 설정할 수가 있습니다. 근데 그 에러 페이지를 따로 설정을 하지 않게 되면은 이런 페이지를 볼 수 있습니다. 근데 이런 페이지에서 보시면은 여기서 뭐 뭐 어드민이라든지 뭐 여러가지 코드들이나 어떤 것이 문제가 되었는지를 일반 사용자에게 보여주기 때문에 이런 것들을 통해 가지고 어떤 공격자는 아 이 어드민에 해당하는 그런 경로가 있구나 이런 것을 알 수가 있습니다. 그래가지고 오류 메시지를 처리할 때는 어떤 오류가 발생했을 때 무조건 일반 사용자에게 그 페이지를 노출을 하면 안되고요. 뭐 따로 뭐 오류 메시지를 노출하지 않는 그런 페이지를 따로 만들어 가지고 그 뭐 400번이나 403번 뭐 이런 500번대 그런 것들은 다 오류와 관련된 리스폰스인데 그런 리스폰스가 발생을 했을 때는 그 서버 개발자가 혹은 소프트웨어 개발자가 직접 설정한 어떤 오류 페이지 공격자가 공격을 하는지 참고할 수 참고할 수 있을 만한 데이터가 없는 페이지로 따로 연결을 시켜줘야 됩니다. 그리고 메시지 뿐만 아니라 여러가지 상황 처리를 할 때도 좀 주의를 해야 되는 부분이 있는데요.

어떤 과정에서 공격자가 예외를 발생하는 입력 공격자뿐만 아니라 어떤 사용자가 데이터를 입력을 했는데 그 데이터가 코드의 개발자 입장에서는 예외에 해당하는 입력이었다. 라고 하면은 그러면은 예외 시에도 예외가 발생하더라도 프로그램이 제대로 잘 동작하도록 설계를 해야 됩니다. 그리고 코딩을 해야 됩니다. 근데 만약에 그것이 없었더라면은 예외 처리가 없는데 예외가 발생했다면 그것은 코드의 문제가 코드가 종료된 등의 문제가 발생할 수 있고요. 만약에 예외 처리를 했어도 적절하게 하지 않았더라면은 예외 발생시 표시되는 데이터나 아니면 경로 등을 표시하게 되면은 이것은 공격자에게 힌트가 될 수가 있습니다. 이거는 인큐립션 과정 중에서 생길 수 있는 암호화 과정 중에서 생길 수 있는 예외 처리를 한 것인데요. 아래쪽 인큐립션에 보게 되면은 스테이트 키의 키 아이디 이런 식으로 들어오게 됩니다. 근데 이 키 아이디가 인덱스를 넘어가게 돼가지고 에러가 발생할 수 있게 되는데요. 그 에러가 발생했을 때 다른 행위를 하지 않고 바로 패스를 해주게 되면은 이 코드에서는 이 기본 키들이 따로 저장이 되어 있기 때문에 그 기본 키를 통해가지고 암호를 하게 됩니다.

근데 이런 디폴트 키를 사용을 해가지고 암호를 하게 되면은 이 디폴트 키가 유출되는 순간 혹은 유출되지 않더라도 이 정도의 어떤 간단한 키 같은 경우에는 공격자가 게싱을 통해가지고 예상을 할 수 있겠죠. 이런 경우에는 어떤 데이터 평문 데이터가 노출되는 등의 문제로 이어질 수가 있는데요. 이렇게 예외 상황이 발생을 했을 때도 문제가 없도록 여기서는 예외가 발생해가지고 오류가 발생을 해도 문제없이 암호화가 진행될 수 있도록 따로 암호화 키를 랜덤으로 생산한다든지 하는 처리를 해주어야 됩니다. 그래서 이렇게 오류가 발생했을 때도 어떤 공격자가 공격을 할 수 있게 취약한 상황에 놓이게 한다든지 그런 것들을 방지를 해야 됩니다. 그리고 이제 에러뿐만 아니라 코드를 작성하는 과정에서도 어떤 논리적 오류가 발생할 수 있는데요. 예를 들면은 널 포인트를 참조를 하는 경우가 있습니다. 공격자나 혹은 일반 사용자가 어떤 객체 값을 설정을 할 수가 있다면 공격자가 객체 값을 널로 지정을 하게 되는 경우에는 널 포인터, 그러니까 널 값을 서버에서 처리를 하는 경우에는 데이터가 있다라고 가정을 했었다면 널 값을 처리하는 경우에 많은 에러가 발생을 할 수가 있습니다.
