SECTION 01

OWASP Top 10의 의미

2강은 취약점 및 공격 유형을 OWASP Top 10을 중심으로 설명한다. OWASP Top 10은 웹 애플리케이션 보안에서 중요한 10가지 취약점을 정리한 목록이다. OWASP 커뮤니티가 개발하고 유지 관리하며, 웹 개발자, 보안 전문가, 소프트웨어를 만드는 사람들에게 널리 사용되는 기준이다.

강의에서는 2017년판과 2021년판을 언급하며, 시간이 지나면서 어떤 취약점이 더 중요해졌고 어떤 취약점 유형이 새로 들어왔는지 확인할 수 있다고 설명한다. 2강에서는 상위 5개 항목을 다룬다.

SECTION 02

Broken Access Control

첫 번째 항목은 Broken Access Control, 즉 접근 제어 실패다. 소프트웨어는 사용자마다 권한을 다르게 부여해야 한다. 카페 서비스라면 일반 사용자는 게시글을 읽고 쓸 수 있지만, 카페 관리자의 설정 변경 권한을 가져서는 안 된다. 그런데 권한을 잘못 주거나 검사를 우회할 수 있으면 사용자가 자기 권한을 넘어 관리자 행위를 할 수 있다.

강의는 보안에서 가장 중요한 원칙 중 하나로 최소 권한의 원칙을 강조한다. 사용자는 필요한 권한만 받아야 하며, 그 이상의 권한을 가져서는 안 된다. 자기 게시글만 수정해야 하는데 다른 사람의 게시글을 수정하거나, 자기 프로필만 봐야 하는데 URL 파라미터를 바꿔 다른 사람의 프로필을 보는 경우가 대표 예시다.

from django.http import HttpResponseForbidden

def user_profile(request, user_id):
    if request.user.id != user_id and not request.user.is_staff:
        return HttpResponseForbidden("권한이 없습니다.")
    return render_profile(user_id)

예시 코드처럼 요청한 user_id가 현재 로그인한 사용자와 같은지 확인하고, 관리자가 아닌데 다른 사람 정보에 접근하려 하면 막아야 한다.

SECTION 03

Cryptographic Failures

두 번째 항목은 Cryptographic Failures다. 암호화 실패 또는 약한 알고리즘 사용으로 민감정보가 노출되는 문제다. 데이터를 평문으로 통신하면 네트워크를 관찰하는 사람이 아이디와 비밀번호 같은 정보를 볼 수 있다. 그래서 암호화가 필요하지만, 오래되거나 약한 알고리즘을 쓰면 암호화했다고 해도 안전하지 않다.

강의에서는 DES, MD5, SHA-1처럼 더 이상 안전하지 않은 알고리즘을 예로 든다. 또 암호화 키를 기본값으로 쓰거나, 부채널 정보처럼 알고리즘 외부의 시간·전력·동작 패턴이 정보를 새게 만드는 경우도 언급한다. HTTP로 id=admin, password=admin 같은 값을 평문 전송하면 누구나 스니핑으로 볼 수 있으므로 민감정보 보호에 실패한다.

SECTION 04

Injection

세 번째 항목은 Injection이다. 공격자가 악의적인 입력을 넣어 서버 측 명령이나 쿼리를 삽입하는 문제다. SQL Injection, Cross-Site Scripting, Command Injection 등이 여기에 포함된다.

강의에서는 로그인 쿼리를 예로 든다. 사용자 아이디와 비밀번호를 문자열 포맷으로 SQL에 직접 끼워 넣으면, 공격자가 입력값 일부를 쿼리 문법으로 해석되게 만들 수 있다. 그러면 원래 의도한 비밀번호 검증이 무력화되고, 쿼리 결과가 항상 참이 되도록 조작될 수 있다. 핵심은 사용자의 입력이 단순 데이터가 아니라 서버가 실행하는 명령 일부가 되어 버리는 것이다.

SECTION 05

Insecure Design

네 번째 항목은 Insecure Design이다. 앞의 항목들이 코드 구현이나 설정 실수에 가까웠다면, 이 항목은 설계 단계에서 생기는 보안 결함을 말한다. 보안 약점은 구현 단계에서만 생기지 않는다. 요구사항 분석에서 잘못된 보안 요구를 만들거나, 설계에서 공격자 모델을 전혀 고려하지 않으면, 코드를 잘 작성해도 설계 자체가 공격을 허용할 수 있다.

예를 들어 인증, 권한, 데이터 흐름, 민감정보 보호 기준을 설계 단계에서 제대로 반영하지 않으면 이후 구현자가 아무리 성실하게 코딩해도 안전한 결과를 만들기 어렵다. 그래서 강의는 설계 단계의 위험 모델링과 보안 통제 설계가 중요하다고 설명한다.

SECTION 06

Security Misconfiguration

다섯 번째 항목은 Security Misconfiguration이다. 보안 기능이 존재하지만 제대로 설정하지 않아 취약해지는 경우다. Django 같은 프레임워크도 여러 보안 기능을 제공하지만, 개발자가 설정하지 않거나 꺼 두면 취약점이 발생할 수 있다.

잘못된 설정위험
불필요한 포트와 기능 활성화공격자가 분석할 표면이 늘어난다.
기본 관리자 계정 사용공개된 기본 아이디와 비밀번호로 접속될 수 있다.
관리자 페이지 기본 경로 사용공격자가 쉽게 관리 화면을 찾을 수 있다.
오류 메시지 노출경로, 프레임워크, 내부 코드 정보가 공격자에게 힌트가 된다.
업그레이드 시 보안 기능 미적용새 버전의 보안 설정을 놓쳐 취약한 상태가 유지된다.

강의는 기본 관리자 페이지 경로와 디폴트 계정·비밀번호 사용으로 공격당하는 사례를 언급하며, 기본값을 그대로 쓰는 것이 실무에서 매우 위험하다고 강조한다.

OWASP TOP10이라고 해서 웹 보안 취약저항들을 많이 다루고 있는 회사에 대해서 잠깐 알아볼 것이고요. 그리고 거기에서 다루고 있는 가장 유의해야 되는 취약점 상위 10개를 모두 소개를 해볼 것입니다. 2강에서는 그 중에서 5개만 먼저 이야기를 할 것입니다. 이 중에서 2강에서는 여기 1번부터 5번에 해당하는 브로컨 액세스 컨트롤부터 다섯 번째에 있는 스크리티 미스 컨피규레이션까지 다룰 예정입니다. 그래서 가장 첫 번째로 탑1에 해당하는 가장 중요한, 가장 위험하다고 알려져 있는 그리고 가장 많이 발생하는 그런 취약점이 무엇이냐 하면 브로컨 액세스 컨트롤입니다. 액세스 컨트롤 같은 경우에는 접근 제어를 이야기를 하는데요. 어떤 소프트웨어를 사용하는 사람들 혹은 웹을 사용하는 사람들에게 그 사람들에게 어떤 권한을 부여를 할 것입니다. 만약에 어떤 카페를 만든다고 하면은 카페 사용자의 권한과 카페 관리자의 권한은 다를 것입니다. 근데 그것이 사용자는 당연히 관리자의 권한에 접근을 하면 안 되겠죠. 근데 그것을 어떤 권한을 잘못 주어가지고 사용자가 사용자의 권한을 벗어나가지고 관리자의 권한까지 할 수 있게 만드는 것. 그런 것이 이제 액세스 컨트롤을 잘 하지 못한 것의 대표적인 예시입니다. 그래서 이 브로컨 액세스 컨트롤이라는 어떤 접근제로 잘못한 그런 취약점이 발생할 수 있는 예시로는 가장 대표적으로는 약 다섯 개 정도로 가져왔는데요.
