SECTION 01

CSRF

CSRF는 공격자가 사용자의 브라우저로 하여금 원치 않는 요청을 보내게 만드는 취약점이다. 강의에서는 게시글의 이미지 태그를 예로 든다. 일반적으로 게시글에 이미지를 올리면 실제 이미지 파일은 웹서버에 저장되고, 게시글에는 이미지 경로가 들어간다. 사용자가 게시글을 열면 브라우저는 그 이미지 경로로 요청을 보낸다.

공격자가 이미지 경로 자리에 악의적인 URL을 넣으면, 나중에 다른 사용자가 해당 게시글을 볼 때 그 사용자의 브라우저가 악의적인 요청을 보내게 될 수 있다. Django는 CSRF 방어 기능을 기본 제공하지만, 설정에서 미들웨어를 빼거나 csrf_exempt 같은 데코레이터로 기능을 끄면 취약점이 생긴다. GitHub에서 내려받은 코드에 이런 설정이 있는지 확인해야 한다.

SECTION 02

SSRF

SSRF는 Server-Side Request Forgery다. 공격자가 직접 접근할 수 없는 URL이라도, 서버는 내부망이나 특정 리소스에 접근할 권한이 있을 수 있다. 공격자는 조작된 요청을 보내 서버가 대신 내부 서버에 요청을 보내게 만들고, 그 응답을 공격자에게 노출시킬 수 있다.

예를 들어 사용자가 입력한 URL을 서버가 그대로 요청하는 기능이 있고, 공격자가 사설 IP 대역의 내부 주소를 넣으면, 서버가 내부의 member.list.json 같은 민감 데이터를 요청해 공격자에게 보여 줄 수 있다. 방어하려면 사용자가 준 주소를 그대로 쓰지 않고, 허용된 URL만 화이트리스트로 처리해야 한다.

import requests
from django.http import HttpResponseBadRequest

ALLOWED_URLS = {
    "https://example.com/latest",
    "https://example.com/public",
}

address = request.GET.get("address", "")
if address not in ALLOWED_URLS:
    return HttpResponseBadRequest("허용되지 않는 주소입니다.")

response = requests.get(address, timeout=3)
SECTION 03

HTTP Response Splitting

HTTP Response Splitting은 응답 헤더를 구성하는 값에 개행 구분자를 삽입해, 서버가 의도하지 않은 새 헤더나 내용을 만들게 하는 취약점이다. 강의에서는 %0d, %0a가 HTTP에서 헤더를 구분하는 문자로 쓰일 수 있다고 설명한다.

예를 들어 사용자 아이디를 응답 헤더에 넣는 코드가 있는데, 그 사용자 아이디에 개행 문자가 들어오면 원래 하나의 헤더 값이어야 할 것이 여러 헤더로 쪼개질 수 있다. 방어하려면 헤더에 들어갈 값에서 개행 문자와 헤더 구분자를 제거하거나, 허용된 문자만 통과시켜야 한다.

SECTION 04

Integer Overflow

Integer Overflow는 허용된 정수 범위를 넘어서는 값이 들어왔을 때 발생한다. 강의에서는 NumPy의 int64를 예로 든다. int64가 표현할 수 있는 범위를 넘어서는 값이나 너무 작은 값이 들어오면 오류가 나거나 프로그램이 멈출 수 있다.

방어 방법은 입력값이 최대·최소 범위를 벗어나는지 먼저 확인하는 것이다. 범위를 벗어나면 함수에 넣지 않고 에러 처리하거나, 업무적으로 허용된다면 최대값·최소값으로 제한한다. 핵심은 위험한 값이 라이브러리 함수에 직접 들어가기 전에 검증해야 한다는 것이다.

SECTION 05

Format String Injection

Format String Injection은 문자열 포맷을 사용자 입력으로 받으면서 생기는 문제다. Python의 format string이나 f-string은 특정 구문을 해석해 값을 꺼낼 수 있다. 공격자가 포맷 문자열을 제어할 수 있으면 원래 보여 주면 안 되는 객체 속성, 키, 세션값, 패스워드 같은 데이터가 노출될 수 있다.

방어하려면 포맷 문자열 자체를 사용자 입력으로 받지 않는다. 개발자가 정한 고정 포맷에 사용자 값을 데이터로만 넣어야 한다. 강의에서는 user_name처럼 필요한 값만 서버 코드에서 지정한 포맷에 넣고, 사용자가 임의의 포맷 구조를 만들지 못하게 해야 한다고 설명한다.

CSRF의 취약점 같은 경우에는 공격자가 이것도 웹서버에 어떤 게시글을 등록을 하는 것인데요. 등록을 할 때 어떤 원래라면 이미지를 등록을 했을 때 웹서버에 이미지가 등록이 될 때는 이미지를 실제로 어떤 게시글에 직접 올라가는 것이 아니라 이미지를 웹서버에 저장을 하고 나서 웹서버에는 웹 게시글에는 그 이미지의 경로가 들어가게 됩니다. 그래서 사용자가 그 게시글을 들어가게 되면 게시글에 써져 있는 이미지의 경로가 어디다라는 것을 보고서 사용자의 컴퓨터에서는 이미지 경로에 해당하는 곳에 어떤 요청을 보내게 되는데요. 근데 이것을 악용해가지고 이미지 태그에 있는 소스에 해당하는 부분에다가 어떤 악의적인 URL을 넣어주게 된다면 나중에 다른 사용자가 그 게시글을 페이지에 요청을 했을 때 그 CSRF의 스크립트에 포함된 이런 악의적인 페이지에 대한 그런 요청이 가게 될 수가 있습니다.

그래서 이 CSRF 같은 경우에도 잔고에서 많이 늘 워낙 유명한 취약점이다 보니까 보안 기능이 설정이 되어 있는데요. 근데 여기에서 잔고의 미들웨어 CSRF를 설정해주는 어떤 세팅이라든지 혹은 CSRF를 제거, CSRF를 기능을 해제해주는 어떤 데코레이터를 사용을 하게 되면 물론 이것을 이제 여러분들이 처음부터 개발할 때는 이런 기능들을 해제하는 기능 같은 경우에는 쓰지 않을 거겠지만 이런 것들이 이제 GitHub에서 보통 소스를 다운받아가지고 사용하는 경우에는 존재할 수 있기 때문에 이런 데코레이터나 아니면은 세팅점 파이에 있는 보안 기능을 제거한 것을 그대로 사용을 하시게 되면은 앞서 말씀드린 것과 같은 CSRF 취약점이 발생할 수 있습니다. 그래서 어떤 코드를 받아와가지고 직접 사용을 할 때라든지 아니면은 실제로 코드를 작성할 때 이런 보안 기능을 여기 있었던 이 보안 기능을 실제로 활성화 해주셔야 되고 그리고 아니면 여기 CSRF.exem 그 데코레이터도 이제 CSRF 기능을 해제하는 것은 사용하지 않도록 해주어야 합니다.

다음은 이제 SSRF입니다. SSRF 같은 경우에는 공격자가 서버로 하여금 서버가 어떤 요청을 공격자가 원하는 요청을 하도록 만드는 것인데요. 일반적으로 공격자에게 허용되지 않은 URL을 서버 같은 경우에는 그 URL에 접속할 수 있는 권한이 존재할 수 있습니다. 그래서 공격자가 어떤 조작된 요청을 보내게 되면은 그 조작된 요청을 웹 서버에서 처리하는 과정에서 어떤 내부 서버에 대한 요청을 보내고 그거에 대한 리스폰스를 공격자에게 노출시키게 되는 그런 상황이 존재할 수 있습니다. 이것이 이제 이런 기법을 보고서 SSRF라고 하는데요. 대표적인 예시로는 이런 것들이 있습니다. 예를 들어서 어떤 사이트에서 커넥트를 해주는 그러니까 그 사이트에서 사이트의 URL에 들어가 있는 이 URL에 그 서버에서 다시 한번 요청을 보내가지고 그 요청을 다시 공격자에게 보여주는 그런 예시가 존재한다고 했을 때 이런 여기 192.168로 시작하는 건 사설 IP잖아요. 그래서 내부망에 존재하는 IP인데 내부망에 있는 member.list.json 뭐 이런 민감정보겠죠.

이런 민감정보들을 공격자 안에 직접 접속할 수가 없으니까 웹서버로 하여금 이런 부분에 한번 요청을 해가지고 그것을 다시 자기한테 표시를 하게 해주는 이런 공격을 사용을 할 수가 있습니다. 그래서 이런 거를 방지를 하기 위해서는 만약에 정말 간단한 예시인데 만약에 서버 측에서 어떤 사용자의 어드레스를 받아가지고 그 어드레스를 기반으로 서버에서 요청을 보내려고 하면 지금 이 사례 같은 경우에는 사용자로부터 받은 어드레스를 그대로 리퀘스트를 보내고 있는데요. 그게 아니라 어떤 어드레스를 받은 것을 그대로 사용을 하는 것이 아니라 사용할 수 있는 그 서버 URL 리스트를 따로 만들어 줘가지고 화이트 리스트 방식으로 예를 들면 여기서는 로컬허스트의 레이티스트 아니면 유저 데이터, 퍼블릭 뭐 이런 식으로 접근할 수 있는 접근할 수 있는 그런 도메인들을 따로 지정을 해가지고 이것에 해당하지 않는 경우에는 모두 다 불가능하도록 다 연결하지 않도록 하는 에러 페이지를 보여주는 그런 식으로 처리를 하는 것이 바람직한 방법입니다. 그리고 이런 식으로 해야만 SSRF로 인한 추약점 발생을 방지할 수 있습니다.

다음은 HTTP 리스폰스 플리팅입니다. 이거 같은 경우에는 이제 브라우저에서 어떤 웹 서버에다가 HTTP 프로토콜을 통해 가지고 어떤 요청을 보내고 나서 다시 리스폰스를 받는 과정에서 중간에 악의적인 입력을 주어 가지고 어떤 의도하지 않았던 그런 그런 리스폰스를 보여주는 것인데요. 이거 같은 경우에는 헤더와 헤더 사이에 존재하는 헤더를 구분해주는 구분자를 추가로 집어 넣어줌으로써 어떤 새로운 헤더를 새로운 헤더를 넣어 줄 수 있게 됩니다. 여기 보시면 컨텐츠 타입에다가 원래 컨텐츠 타입에 들어가야 되는 텍스트랑 HTML뿐만 아니라 뒤에 어떤 어떤 특수문자를 넣고 나서 테스트 스크립트에 해당하는 데이터를 넣어주게 되었는데요. 이 %d, %a가 HTTP 프로토콜에서 헤더를 구분해주는 구분자에 해당하므로 여기 한번 구분해놓고서 구분자가 하나 더 있었기 때문에 또 이것을 새로운 헤더를 인식하게 만들게 됩니다. 그래서 이것에 대해서 어떤 응답이 갔을 때는 이런 식으로 테스트라고 되어 있는 해당하는 부분에는 아예 다른 새로운 헤더 혹은 새로운 컨텐츠로 처리를 하게 되는 것이죠.
