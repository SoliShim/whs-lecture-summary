입력 검증이 중요한 이유

4강부터는 시큐어 코딩 기법을 다룬다. 그중 첫 주제는 입력 데이터 검증 및 표현이다. 대부분의 해킹 공격은 악의적인 입력을 주는 방식으로 이루어진다. 개발자가 어떤 입력을 허용하고 어떤 입력을 허용하지 않을지 잘 필터링하는 것만으로도 많은 공격을 막을 수 있다.

그래서 입력 데이터 검증은 보안 기능, 시간 및 상태, 에러 처리, 코드 오류, API 오용 같은 다른 분류보다 큰 비중을 가진다. 사용자의 입력은 항상 신뢰하지 않고, 의도한 형태인지 확인해야 한다.

SQL Injection

웹 서버는 로그인, 게시글 작성, 댓글 작성, 수정 같은 기능을 수행할 때 데이터베이스와 상호작용한다. 공격자가 조작된 요청을 보내면 개발자가 의도한 DB 쿼리와 다른 방식으로 데이터베이스가 동작할 수 있다. 이것이 SQL Injection이다.

문제는 사용자의 입력값을 검증하지 않고 SQL 문자열 안에 직접 끼워 넣을 때 발생한다. 입력값이 단순한 이름이 아니라 SQL 문법 일부로 해석되면, 원래 특정 게시글만 수정해야 했던 쿼리가 전혀 다른 결과를 만들 수 있다.

방어 방법은 사용자 입력을 SQL 문자열에 직접 붙이지 않고, 쿼리에는 자리 표시자를 두며 실제 값은 별도 인자로 넘기는 것이다. 그러면 공격자가 특수한 문자열을 넣어도 쿼리 문법이 아니라 데이터 값으로 처리된다.

name = request.POST.get("name", "")
content_id = request.POST.get("content_id", "")

cursor.execute(
    "UPDATE posts SET name = %s WHERE content_id = %s",
    [name, content_id],
)

Code Injection

Code Injection은 사용자의 입력이 서버에서 코드처럼 실행되는 문제다. Python의 eval()처럼 문자열을 코드로 해석하는 함수는 편리할 수 있지만, 사용자 입력을 검증 없이 넣으면 공격자가 보낸 코드가 서버에서 실행될 수 있다.

강의에서는 time.sleep() 같은 코드를 넣으면 웹서버가 멈추는 예를 언급한다. 더 악의적인 입력이라면 백도어나 정보 탈취 코드도 실행될 수 있다. 방어하려면 애초에 사용자 입력을 코드로 실행하지 않고, 정말 필요한 경우에도 영문·숫자 등 의도한 데이터만 허용해야 한다.

message = request.POST.get("message", "")
if not message.isalnum():
    return HttpResponseBadRequest("허용되지 않는 입력입니다.")

# 사용자의 문자열을 코드로 실행하지 않는다.
process_message(message)

Path Traversal

웹 서버는 파일 업로드와 다운로드 기능을 제공할 수 있다. 만약 다운로드 URL이 파일 이름과 경로를 사용자 입력으로 받아 처리한다면, 공격자는 ..나 / 같은 경로 문자를 조작해 원래 허용된 디렉터리 밖의 파일을 읽으려 할 수 있다. Linux의 /etc/passwd 같은 민감 파일을 노리는 방식이 대표적이다.

강의에서는 파일 확장자가 txt나 csv인지 확인하는 것만으로는 충분하지 않다고 설명한다. 파일명 자체가 상위 경로를 가리키지 않는지, 최종 경로가 허용된 디렉터리 안에 있는지, 허용 확장자인지 함께 확인해야 한다.

from pathlib import Path
from django.http import HttpResponseBadRequest

BASE_DIR = Path("/srv/app/data").resolve()

filename = request.GET.get("file", "")
target = (BASE_DIR / filename).resolve()

if BASE_DIR not in target.parents or target.suffix not in {".txt", ".csv"}:
    return HttpResponseBadRequest("허용되지 않는 파일입니다.")

return FileResponse(open(target, "rb"))

XSS

XSS는 HTML 태그 사이에 공격 스크립트를 넣어, 다른 사용자의 브라우저에서 악의적인 JavaScript가 실행되게 만드는 취약점이다. 게시글이나 프로필 URL 같은 입력값이 웹페이지에 그대로 들어가면 공격자는 태그를 닫고 script 태그를 새로 열어 악성 코드를 삽입할 수 있다.

Django는 유명한 취약점인 XSS를 기본적으로 막기 위한 기능을 제공한다. 하지만 온라인에서 받은 코드를 그대로 쓰거나, 특별한 이유로 mark_safe 같은 함수를 사용해 escaping을 끄면 XSS가 발생할 수 있다. 코드에 이런 함수가 있는지 확인하고, 정말 필요한 경우가 아니라면 제거해야 한다.

Command Injection

Command Injection은 사용자의 입력값이 운영체제 명령어로 실행되는 문제다. Python의 os.system() 같은 함수에 사용자 입력을 그대로 넣으면, 공격자는 ls, dir, set 같은 명령어 또는 더 위험한 명령을 서버에서 실행하게 만들 수 있다.

강의 예시는 애플리케이션 이름을 받아 실행하려는 코드다. 개발자는 특정 프로그램 실행만 의도했지만, 공격자가 다른 명령 문자열을 넣으면 서버 명령 실행으로 이어질 수 있다. 방어하려면 개발자가 의도한 명령 또는 프로그램 이름만 허용하는 화이트리스트를 두고, 그 외 입력은 에러로 처리해야 한다.

파일 업로드 취약점

파일 업로드 취약점은 게시판이나 업로드 기능에 웹셸 같은 악의적 파일을 올릴 수 있을 때 발생한다. 업로드 파일의 종류, 크기, 개수, 이름, 확장자를 전혀 검증하지 않으면 공격자는 서버가 실행할 수 있는 악성 파일을 업로드해 웹서버를 공격할 수 있다.

방어하려면 개발자가 의도한 파일만 업로드되도록 제한해야 한다. 예를 들어 이미지만 받는 기능이라면 MIME 타입과 확장자를 확인하고, 파일 크기 제한을 두며, 너무 많은 파일을 올리지 못하게 해야 한다. 업로드한 파일을 실행 가능한 위치에 두지 않는 것도 중요하다.

ALLOWED_CONTENT_TYPES = {"image/png", "image/jpeg"}
MAX_SIZE = 2 * 1024 * 1024

uploaded = request.FILES["file"]
if uploaded.content_type not in ALLOWED_CONTENT_TYPES:
    return HttpResponseBadRequest("이미지 파일만 업로드할 수 있습니다.")
if uploaded.size > MAX_SIZE:
    return HttpResponseBadRequest("파일이 너무 큽니다.")

save_uploaded_image(uploaded)
