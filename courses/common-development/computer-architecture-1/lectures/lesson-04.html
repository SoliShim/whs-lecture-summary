왜 CPU를 프로그램 실행 과정으로 배우는가

멘토는 CPU를 컴퓨터의 “꽃”처럼 표현한다. CPU를 이해하려면 먼저 프로그램이 실행되는 과정을 봐야 한다. 이 흐름은 컴퓨터 구조뿐 아니라 리버스 엔지니어링, 어셈블리어, 시스템 해킹, 운영체제 수업에서 계속 다시 나온다.

C 언어를 공부할 때 막연히 문법만 보면 재미없을 수 있지만, 내가 작성한 코드가 실제로 어떤 명령어가 되고 CPU에서 어떻게 실행되는지 알면 C가 왜 중요한지 이해하기 쉬워진다. 리버싱은 기계어와 어셈블리어를 보고 원래 소스 코드의 의도를 거꾸로 추정하는 작업이므로, 이 흐름이 특히 중요하다.

소스 코드에서 기계어까지

사람이 읽기 쉬운 C 소스 코드는 컴파일 과정을 거치며 어셈블리어에 가까운 형태를 지나고, 최종적으로 CPU가 해석할 수 있는 기계어 또는 바이너리가 된다. 강의에서는 컴파일러, 링커 같은 세부 단계가 있지만 지금은 큰 흐름을 잡는 것이 중요하다고 설명한다.

CPU가 해석하는 0과 1의 명령

예를 들어 mov 명령은 source 값을 destination으로 옮기는 명령으로 이해하면 된다. destination과 source에는 레지스터, 메모리 주소, 상수 등이 올 수 있다.

mov eax, 10        ; eax 레지스터에 10을 넣는다
mov [ebp-4], eax   ; eax 값을 메모리 위치에 저장한다

기계어는 사람이 직접 읽기 어려워서 보통 16진수 형태로 확인한다. 이 16진수 기계어가 CPU 아키텍처의 명령어 표에 따라 어셈블리 명령으로 해석된다.

명령어 사이클: Fetch, Decode, Execute

CPU가 하는 가장 기본적인 일은 Fetch, Decode, Execute의 반복이다. Fetch는 명령어를 가져오는 단계, Decode는 가져온 명령어가 무엇인지 해석하는 단계, Execute는 실제로 명령을 수행하는 단계다.

Fetch

메모리에서 다음 명령어를 가져온다.

Decode

명령어가 어떤 동작을 의미하는지 해석한다.

Execute

해석한 명령을 실제로 실행한다.

멘토는 CPU가 이 점에서 “멍청하지만 천재적”이라고 표현한다. CPU가 할 수 있는 일은 정해진 명령을 가져오고 해석하고 실행하는 것뿐이지만, 그것을 엄청나게 빠르게 반복하기 때문이다. GHz 같은 클럭 수치는 초당 얼마나 많은 사이클을 처리할 수 있는지와 연결된다.

확장된 CPU 실행 단계

기본 세 단계만으로 설명하기 부족할 때는 실행 단계를 더 나눠서 본다. 강의에서는 Fetch, Decode, Execute에 더해 Memory Access와 Write Back을 언급한다. 예를 들어 mov 명령 하나를 수행하려 해도 source 값을 읽고, destination 위치를 확인하고, 결과를 다시 저장해야 할 수 있다.

Fetch

명령어를 가져온다.

Decode

명령어와 피연산자를 해석한다.

Execute

연산을 수행한다.

Memory Access

필요하면 메모리를 읽거나 쓴다.

Write Back

결과를 레지스터나 메모리에 반영한다.

문제는 CPU가 매우 빠른 데 비해 메모리 접근은 상대적으로 느리다는 점이다. 그래서 뒤에서 배우는 메모리 계층 구조와 캐시가 중요해진다.

CPU 성능을 높이는 생각들

CPU는 단순히 클럭만 높이는 방향으로 발전하지 않는다. 강의에서는 파이프라이닝, 분기 예측, 명령어 수준 병렬성 같은 개념을 언급한다. 지금 단계에서 세부 구현을 외울 필요는 없지만, CPU가 놀지 않게 만들고 다음에 필요한 작업을 미리 준비하려는 최적화라고 이해하면 된다.

  • 파이프라이닝: 한 명령의 모든 단계가 끝날 때까지 기다리지 않고, 서로 다른 명령의 다른 단계를 겹쳐 처리한다.
  • 분기 예측: 반복문이나 조건문에서 다음에 어디로 갈지 예측해 미리 준비한다.
  • 명령어 수준 병렬성: 서로 영향을 주지 않는 명령들을 가능한 한 동시에 처리하려는 방향이다.

멘토는 웹 개발이나 앱 개발을 하더라도 성능 문제가 생기면 결국 메모리 접근, CPU 작업, 입출력 병목을 생각하게 된다고 말한다. 그래서 컴퓨터 구조 지식은 특정 분야에만 갇힌 지식이 아니다.

리버싱과 보안 관점

프로그램은 실행되기 위해 반드시 CPU가 이해할 수 있는 형태의 명령을 남긴다. 그래서 완성된 프로그램을 분석해 어떤 명령어가 있는지 확인하고, 그 의미를 거꾸로 추적하는 리버스 엔지니어링이 가능하다.

방어자는 패킹, 압축, 난독화 같은 기법으로 분석을 어렵게 만들 수 있다. 하지만 프로그램이 실행되어야 하는 이상 완전히 읽을 수 없게 만드는 것은 어렵다. 멘토는 보안의 목표를 “절대 뚫리지 않게 만들기”보다는 공격자가 분석하는 데 드는 시간과 비용을 크게 늘리는 것으로 설명한다.

CISC와 RISC

CPU 아키텍처마다 명령어 집합이 다르다. 어떤 CPU가 어떤 명령을 이해하는지에 따라 실행 가능한 프로그램이 달라진다. 그래서 프로그램 다운로드 페이지에서 x86, x64, ARM 같은 선택지가 나타난다.

구분강의의 설명이해 포인트
CISC복잡하고 많은 명령어를 가진 구조하나의 명령이 더 큰 일을 할 수 있어 프로그래머에게 편할 수 있다.
RISC줄이고 단순화한 명령어 집합명령은 단순하지만 빠르고 효율적으로 설계하기 좋다.

강의에서는 전통적으로 CISC는 복잡하고 느리며, RISC는 단순하고 빠르다는 식으로 배웠지만, 현대 CPU는 둘 다 매우 발전해서 단순 비교만으로 우열을 말하기 어렵다고 덧붙인다. ARM은 RISC 흐름에서 강력하게 성장했고, Apple Silicon과 모바일 환경을 이해할 때 중요한 이름이다.

CPU 내부 구성

CPU 내부는 크게 제어 장치, ALU, 레지스터, CPU 내부 연결 구조로 생각할 수 있다. 제어 장치는 명령의 흐름을 조정하고, ALU는 산술 및 논리 연산을 수행한다. 레지스터는 CPU 안에 있는 아주 작고 빠른 저장 공간이다.

구성 요소역할
Control Unit명령어 흐름을 제어하고 필요한 동작을 지시한다.
ALU덧셈, 뺄셈, 곱셈, 나눗셈, AND, OR, XOR 같은 산술·논리 연산을 처리한다.
RegistersCPU가 바로 사용할 값을 임시로 담는 매우 빠른 저장 공간이다.
InterconnectionCPU 내부 구성 요소가 데이터를 주고받는 연결 통로다.

멘토는 곱셈도 2의 거듭제곱과 관련된 경우 비트 시프트로 빠르게 처리할 수 있다는 식의 감각을 언급한다. 결국 CPU 내부 연산은 비트와 논리 연산의 조합으로 이어진다.

싱글코어와 멀티코어

지금까지의 설명은 하나의 코어가 명령을 실행하는 그림에 가깝다. 하지만 하드웨어가 발전하면서 하나의 프로세서 안에 여러 코어를 넣을 수 있게 되었다. 한 코어를 하나의 CPU처럼 생각하면, 멀티코어 프로세서는 여러 실행 단위를 한 칩 안에 담은 구조로 볼 수 있다.

마무리에서 멘토는 프로그램 실행 과정이 가장 중요하다고 다시 강조한다. 나중에 운영체제나 시스템 해킹을 배우면 C 소스 코드가 어셈블리어와 기계어로 어떻게 나뉘는지 직접 보게 될 수 있다. 이 과정이 재미있다면 시스템 해커 쪽이 잘 맞을 수 있고, 그렇지 않더라도 CPU가 이런 방식으로 실행된다는 사실은 성능 문제를 이해하는 데 도움이 된다.
