데몬 서비스와 systemd

리눅스에는 사용자가 직접 실행한 프로그램 외에도 백그라운드에서 계속 동작하는 서비스가 많다. SSH, 웹 서버, 데이터베이스 서버처럼 시스템 시작과 함께 올라오고 계속 요청을 기다리는 프로그램을 데몬 서비스로 이해하면 된다.

과거에는 init 프로세스가 이런 서비스를 띄웠고, 지금 우분투에서는 systemd가 그 역할을 한다. systemd가 서비스, 소켓, 마운트, 부팅 타겟을 관리하고, 사용자는 systemctl로 제어한다.

런레벨과 타겟

고전적인 init 시스템에서는 runlevel 0, 1, 3, 5, 6 같은 숫자로 부팅 상태를 표현했다. 예를 들어 0은 종료, 1은 복구 모드, 3은 멀티유저 텍스트 모드, 5는 그래픽 모드처럼 이해할 수 있다. systemd에서는 이를 poweroff.target, rescue.target, multi-user.target, graphical.target처럼 글자로 표현한다.

기초 강의에서는 타겟을 깊게 다루지 않지만, systemd가 단순히 서비스 하나만 관리하는 것이 아니라 시스템 부팅 상태 전체를 관리한다는 점은 기억해야 한다.

systemctl 기본 명령

systemctl status sshd처럼 서비스 상태를 볼 수 있다. 정식 이름은 sshd.service지만, .service는 생략해도 동작한다. 서비스 제어에는 관리자 권한이 필요하므로 start, stop, restart 같은 명령에는 보통 sudo가 필요하다.

systemctl status ssh
sudo systemctl start nginx
sudo systemctl stop nginx
sudo systemctl restart nginx
systemctl status nginx

리눅스 명령이 성공했을 때는 별도 메시지가 없는 경우가 많다. 오류가 나오지 않았으면 정상적으로 처리되었다고 보고, 상태 확인 명령으로 다시 확인한다.

enable과 disable

start와 stop은 현재 실행 상태를 바꾼다. 반면 enable과 disable은 부팅 시 자동 시작 여부를 바꾼다. 따라서 현재 active이면서 disabled일 수도 있고, inactive이면서 enabled일 수도 있다.

상태 조합의미
active + enabled지금 실행 중이고, 재부팅 후에도 자동 실행된다.
inactive + enabled지금은 꺼져 있지만, 재부팅하면 자동 실행된다.
active + disabled지금은 켜져 있지만, 재부팅 후 자동 실행되지는 않는다.
inactive + disabled지금도 꺼져 있고, 재부팅해도 자동 실행되지 않는다.
 
sudo systemctl enable nginx
sudo systemctl disable nginx
systemctl status nginx

journalctl과 로그

서비스가 실행되며 남긴 로그는 journalctl로 확인할 수 있다. 특정 서비스의 로그를 보거나, 특정 시간 이후의 로그를 볼 수 있다. 로그는 장애 분석에 중요하지만 계속 쌓이면 디스크를 차지하므로 관리가 필요하다.

로그에는 emergency, alert, critical, error, warning, notice, info, debug처럼 심각도 레벨이 있다. 단순 정보인지, 실제 장애인지 구분해서 읽을 수 있어야 한다.

journalctl -u nginx
journalctl -u nginx --since today
journalctl -xe

Nginx 실습 흐름

실습에서는 sudo apt install nginx로 웹 서버를 설치하고 systemctl status nginx로 상태를 확인한다. curl localhost나 브라우저에서 localhost를 열어 웹 서버가 응답하는지 본다. Nginx를 stop하면 curl과 브라우저 요청이 실패하고, 다시 start하면 응답이 돌아온다.

sudo apt install nginx
systemctl status nginx
curl localhost
sudo systemctl stop nginx
curl localhost
sudo systemctl start nginx
curl localhost

강의 전체 정돈본

아래 내용은 STT 원문 흐름을 유지하면서 문장과 문단을 나눠 읽기 쉽게 정리한 전체 강의 본문이다. 세부 설명을 건너뛰지 않고 따라가야 하는 강의는 이 영역까지 함께 읽으면 된다.

운영체제 파트를 맡고 있는 박수현 멘토입니다. 네, 이번은 운영체제에 대한 7번째 시간으로 데몬 서비스 관리에 대해 배워보도록 하겠습니다. 네, 그리고 제가 처음 강의 때 총 우리가 9강까지 진행할 거라고 말씀을 드렸었는데 전체적으로 수업을 진행하다 보니까 분량이 꽤 많이 지금 이제 영상 분량이 나오고 있어서 나머지 8강과 9강은 제가 이후에 할 이제 오프라인 수업 때 그때 다룰 예정이고요. 그래서 여러분들이 온라인으로 수업하시는 거는 7강까지 그래서 이번 강의를 끝으로 내 온라인 강의는 마치게 되는 거라고 이해를 하시면 되겠습니다. 자, 그래서 이전 시간에 이제 우리가 패키지 설치까지 달아봤고요. 그럼 이제 우리가 설치한 다양한 그런 데몬 서비스를 갖다가 이제 관리하는 그런 방법들을 살펴보도록 하겠습니다. 네, 일단 데몬 서비스란 무엇이냐? 네, 이거부터 알아야 될 것 같은데요. 네, 데몬 서비스라고 하는 것은 사용자가 직접적으로 제어하거나 하지 않고 백그라운드에서 돌면서 여러 작업을 하는 프로그램을 데몬이라고 부릅니다. 네, 서비스란 이건 이제 주로 서버 클라이언트 모델에서 출발을 해서 사용자의 요청에 따라서 응답해주는 다양한 프로그램들을 의미하고요. 그래서 많은 서비스가 주로 데몬 형태로 구동이 되어 있습니다. 그래서 이렇게 애매하게 이렇게 겹치는 바운더리들을 많이 가지고 있고요. 그래서 그럼 이제 데몬 서비스에 대한 예시는 웹 서버, 백그라운드에서 혼자 돌면서 언제 요청할지 모르는 그런 불득정 다수의 사용자의 요청이 왔을 때 그때 반응하고 응답을 해주는 거잖아요. 네, 또는 유사한 걸로 파일 서버도 있겠죠. FTP 서버. 또는 전혀 다른 이제 꼭 네트워크와 외부 사용자가 아닌 이렇게 나의 로컬 환경 내에서 내 프린터 서버가 이제 혼자 조용히 돌고 있다가 사용자가 다큐먼트 프로그램에서 이렇게 프린트를 눌렀을 때 그때 사용자의 요청을 받아서 프린터라는 하드웨어에 나의 문서 파일을 갖다가 전달해주는 그런 서비스 데몬들도 존재하게 됩니다. 이제 이렇게 다양한 서비스가 있고 이런 것들은 비단 리눅스뿐만이 아니고 윈도우 내도 이렇게 엄청나게 많은 서비스 데몬들이 동작하고 있습니다. 그래서 여러분들이 사용하시는 그 윈도우도 우클릭해갖고는 그 이제 작업관리자를 살펴보시면 이렇게 서비스를 볼 수 있는 메뉴가 있고요. 그걸 살펴보시면은 거의 한 100개 가까이 되는 많은 서비스가 이렇게 백그라운드에서 돌고 있구나라고 하는 것을 아실 수가 있습니다. 네 그래서 우분투에서는 또는 이렇게 고전적으로 다양한 리눅스 배포판 내에서는 시스템5라고 하는 이렇게 데몬 관리자가 있었었고요. 그래서 이제 시스템5는 서비스라고 하는 요인제 명령어를 통해서 다양한 데몬 서비스를 관리할 수 있었습니다. 그래서 지금도 여러분들이 검색해보시면 이 서비스라는 명령어를 많이 많이 보실 수가 있습니다. 여러 명의 블로그에서. 네 하지만 그런 것들이 나오는 블로그는 되게 올드한 옛날 블로그인 거예요. 지금도 쓸 수는 있지만 사실은 서비스라는 명령어가 단종되고 쓰이지 않는지는 꽤 오래됐습니다. 그래서 오래된 블로그고 이런 개념과 원리와 시대의 흐름을 갖다가 잘 이해하지 못하는 블로그이기 때문에 과감하게 종료 버튼을 누르시고 네 그런 블로그 내용들은 그리고 좀 더 좋은 내용을 찾으셔서 네 다음 페이지에서 배우린 이제 시스템ctl 이라는 명령어로 소개되어 있는 블로그들이 좋은 블로그인 거예요. 그래서 지금은 과거 호환성 유지 때문에 서비스를 아직도 쓸 수가 있다
