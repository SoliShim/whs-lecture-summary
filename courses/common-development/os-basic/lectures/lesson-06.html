SECTION 01

Windows식 개인용 PC 관점과 Linux식 멀티유저 관점

강의는 Windows와 Linux의 출발점 차이에서 시작한다. Windows는 MS-DOS와 초기 Windows 시절부터 개인용 PC, 즉 한 사람이 자기 컴퓨터를 쓰는 환경을 중심으로 발전했다. 그래서 과거에는 별도의 사용자 계정이나 권한 구분이 약했고, “내가 사용자이면서 곧 관리자”라는 관점이 자연스러웠다. 지금의 Windows는 계정과 권한 기능을 갖추었지만, 역사적으로는 개인용 운영체제의 성격이 강했다.

반면 Linux는 Unix 계열 철학을 이어받았다. Unix는 한 사람이 혼자 쓰는 PC가 아니라, 여러 사용자가 메인프레임이나 서버에 동시에 붙어 작업하는 환경에서 출발했다. 여러 사람이 같은 시스템을 쓰는데 아무나 프로그램을 설치하고, 아무나 삭제하고, 아무 때나 재부팅하면 장애와 인적 오류가 바로 발생한다. 그래서 리눅스는 처음부터 사용자와 관리자를 분리하고, 파일과 디렉토리마다 소유자(owner), 그룹(group), 기타 사용자(other)에 대한 권한을 둔다.

강사가 반복해서 강조한 핵심 문장은 “사용자로서의 나와 관리자로서의 나는 다르다”이다. 내가 설치한 Ubuntu라도 일반 사용자로 로그인한 나는 시스템 전체를 마음대로 바꿀 수 있는 관리자가 아니다. 필요한 순간에만 관리자 권한을 빌려야 하고, 그 권한으로 무엇을 하는지 이해해야 한다.

SECTION 02

root, superuser, sudo: 강력한 권한을 빌려 쓰는 방식

리눅스의 최상위 관리자는 root다. root는 superuser라고도 부르며, 그 위에 더 높은 관리자는 없다. root는 권한 검사를 사실상 뛰어넘어 모든 파일을 읽고, 쓰고, 실행하고, 패키지를 설치하고, 시스템 설정을 바꿀 수 있다. 이 힘은 편리하지만 위험하다. 정상 파일인 줄 알고 실행한 것이 악성코드라면, root 권한으로 실행되는 순간 시스템 전체와 다른 사용자 계정까지 피해를 줄 수 있다.

Ubuntu는 초보자와 일반 사용자가 root 계정을 남용하는 문제를 줄이기 위해 root 직접 로그인을 기본적으로 막고, sudo를 사용해 필요한 명령 하나에 대해서만 관리자 권한을 빌리게 한다. sudo는 superuser 권한으로 do, 즉 “관리자 권한으로 수행한다”는 의미로 이해하면 된다.

whoami
id
groups
sudo whoami
sudo id

whoami는 현재 명령을 실행하는 사용자를 보여 주고, id는 UID, GID, 소속 그룹을 보여 준다. 설치할 때 만든 첫 계정 user1은 보통 UID 1000, GID 1000을 가지며, sudo, adm, cdrom, netdev, sambashare 같은 여러 그룹에 들어 있다. 강사는 이 계정이 “유일한 사용자이면서 설치를 수행한 계정”이기 때문에 여러 장치와 관리 기능에 접근할 수 있는 그룹을 기본으로 가진다고 설명한다.

sudo whoami를 실행하면 그 명령을 수행하는 동안에는 결과가 root로 나온다. 이어서 다시 whoami를 치면 user1로 돌아온다. 즉 sudo는 내가 영구히 root가 되는 명령이 아니라, 그 순간만 root 권한을 빌려 오는 장치다. 그래서 권한이 없다고 나올 때마다 습관적으로 sudo를 붙이는 것은 좋지 않은 습관이다. 먼저 “왜 권한이 없는지”, “정말 관리자 권한이 필요한 작업인지”, “내가 지금 어떤 파일을 건드리는지”를 생각해야 한다.

sudo를 아무나 사용할 수 있는 것도 아니다. /etc/sudoers 파일은 어떤 사용자나 그룹이 sudo를 쓸 수 있는지, 어떤 방식으로 쓸 수 있는지 관리한다. 일반 사용자로 cat /etc/sudoers를 실행하면 권한이 없어 볼 수 없고, 관리자 권한으로 sudo cat /etc/sudoers를 해야 확인할 수 있다. 파일 안에서 %로 시작하는 항목은 그룹을 의미한다. 이 파일의 세부 문법은 복잡하므로 기초반에서는 “sudo 권한도 별도 정책 파일로 관리된다”는 점만 확실히 잡으면 된다.

cat /etc/sudoers
sudo cat /etc/sudoers
SECTION 03

/etc/passwd, /etc/shadow, /etc/group 읽기

리눅스의 로컬 계정 정보는 대표적으로 세 파일에 나뉘어 저장된다. /etc/passwd는 사용자 계정 목록과 기본 정보를 담고, /etc/shadow는 암호 해시와 암호 정책을 담으며, /etc/group은 그룹 정보를 관리한다.

cat /etc/passwd
cat /etc/shadow
sudo cat /etc/shadow
cat /etc/group

/etc/passwd라는 이름 때문에 비밀번호가 그대로 들어 있다고 오해하기 쉽다. 과거에는 이 파일 안에 암호 해시가 있었지만, /etc/passwd는 여러 프로그램이 사용자 정보를 확인해야 하므로 누구나 읽을 수 있었다. 해시가 평문은 아니더라도, 공격자가 해시를 얻으면 사전 공격이나 크래킹 도구로 약한 패스워드를 맞힐 수 있다. 그래서 민감한 암호 해시는 /etc/shadow로 분리되었고, 일반 사용자는 shadow를 읽지 못한다.

/etc/passwd 한 줄은 콜론(:)으로 구분된다. 필드는 순서대로 사용자명, 패스워드 자리표시자, UID, GID, 사용자 이름 또는 설명, 홈 디렉토리, 로그인 셸이다. 현재는 실제 암호가 shadow로 빠져 있으므로 패스워드 칸에는 보통 x가 들어간다. 예를 들어 user1:x:1000:1000:user1,,,:/home/user1:/bin/bash 같은 줄은 user1의 UID와 GID가 1000이고, 홈 디렉토리는 /home/user1, 로그인 셸은 /bin/bash라는 뜻이다.

UID 0은 root다. 1~99 사이의 낮은 번호는 전통적으로 미리 정의된 시스템 계정에 쓰이고, 100~999는 관리용 또는 시스템 서비스 계정에 활용될 수 있도록 남겨 둔 범위로 설명한다. 실제 사용자가 로그인해 작업하는 계정은 보통 1000번부터 시작한다. 화면의 여러 계정 중 /usr/sbin/nologin을 로그인 셸로 가진 계정은 사람이 터미널로 로그인해 쓰는 계정이 아니라, 프린터, 메일, 데몬 같은 시스템 서비스 실행을 위한 계정으로 보면 된다.

/etc/shadow에는 사용자명, 암호 해시, 마지막 암호 변경일, 최소 사용일, 최대 사용일, 만료 경고일, 비활성 기간, 계정 만료일 같은 정보가 들어간다. 강의에서는 마지막 변경일처럼 보이는 18357 같은 숫자를 설명한다. 리눅스의 시간 계산은 epoch, 즉 1970년 1월 1일 00:00:00 UTC를 기준으로 한다. 1970년 1월 1일에 18,357일을 더하면 2020년 4월 5일이 되므로, shadow의 날짜 숫자도 계산 가능한 값이라는 점을 보여 준다.

root 계정의 암호 칸에 느낌표(!)가 있으면 암호 로그인이 잠겨 있다는 뜻이다. Ubuntu가 root 직접 로그인을 기본적으로 막는 흐름과 연결된다. 일반 사용자 암호는 평문이 아니라 단방향 해시로 저장되며, 강의 화면은 $1$ MD5, $2a$ Blowfish, $5$ SHA-256, $6$ SHA-512 같은 식별자를 예로 보여 준다. 여기서 중요한 것은 “암호를 저장하더라도 평문이 아니라 해시로 저장하고, 그 해시 파일 자체도 보호한다”는 구조다.

SECTION 04

사용자와 그룹 만들기, 바꾸기, 삭제하기

새 사용자 계정은 아무나 만들 수 없다. 회사에서 신입사원이 마음대로 계정을 만들 수 있다면, 퇴사 전에 백도어 계정을 만들거나 권한 추적을 어렵게 만들 수 있다. 그래서 계정 생성은 관리자 권한으로 수행한다.

adduser user2
sudo adduser user2
cat /etc/passwd
sudo cat /etc/shadow

일반 사용자로 adduser user2를 실행하면 실패하고, sudo adduser user2로 실행해야 한다. 생성 과정에서는 user2의 홈 디렉토리를 만들고, 기본 그룹을 만들고, 암호와 사용자 정보를 입력한다. 이후 /etc/passwd를 보면 user2 항목이 추가되고, /etc/shadow를 관리자 권한으로 보면 user2의 암호 해시와 날짜 정보도 생긴다.

sudo passwd user2
sudo chage -l user2
sudo chage user2
sudo chage -E 2026-12-31 -m 1 -M 90 -W 7 user2

passwd는 실제 암호를 바꾸는 명령이다. 반면 chage는 암호의 정책을 바꾼다. 예를 들어 외주 직원 계정이 연말까지만 유효해야 한다면 계정 만료일을 지정할 수 있고, 암호를 최소 며칠 사용해야 하는지, 최대 며칠마다 바꿔야 하는지, 만료 며칠 전부터 경고할지도 지정할 수 있다. chage -l user2는 현재 정책을 목록으로 보여 준다.

sudo deluser user2
sudo deluser --remove-home user2

deluser user2는 사용자 계정을 삭제한다. 하지만 기본 삭제는 홈 디렉토리까지 항상 지우지는 않는다. 강사는 새 직원을 받거나 퇴사자를 처리할 때 그 사람의 문서와 작업물을 바로 삭제하면 안 되는 경우가 많기 때문에, 홈 디렉토리 보존 여부가 중요하다고 설명한다. 홈 디렉토리까지 지우려면 --remove-home 같은 옵션을 명시해야 한다.

sudo addgroup group1
sudo delgroup group1

그룹도 별도로 만들고 삭제할 수 있다. 그룹은 파일 권한에서 owner와 other 사이에 있는 중간 범주다. 같은 팀, 같은 역할, 같은 서비스 계정을 묶어 권한을 부여할 때 사용한다.

sudo usermod -a -G sudo user2
id user2
sudo deluser user2 sudo
id user2

usermod -a -G sudo user2는 user2를 sudo 그룹에 추가한다. 여기서 -a는 append, 즉 기존 그룹을 유지한 채 새 그룹을 추가한다는 뜻이다. -G만 잘못 쓰면 기존 보조 그룹 구성이 바뀔 수 있으므로 주의해야 한다. 그룹에서 제거할 때는 Ubuntu에서 sudo deluser user2 sudo처럼 쓸 수 있다. 강의 화면의 한국어 메시지는 “user2 그룹에서 sudo 사용자 제거”처럼 순서가 어색하게 번역되어 보이는데, 실제 영어 의미는 “sudo 그룹에서 user2 사용자를 제거한다”에 가깝다. 기능은 맞고 번역만 헷갈리는 사례로 이해하면 된다.

강사는 실무에서는 로컬 계정만으로 모든 사용자를 관리하지 않고 LDAP이나 Active Directory 같은 중앙 인증 체계를 쓰는 경우도 많다고 덧붙인다. 그래도 로컬 파일과 기본 명령을 이해해야 중앙 인증도 제대로 이해할 수 있다.

SECTION 05

파일 권한: owner, group, other와 rwx

파일 권한은 ls -l의 첫 열에서 확인한다. 첫 글자는 파일 유형이다. -는 일반 파일, d는 디렉토리, l은 심볼릭 링크다. 그 뒤 9글자는 세 글자씩 나뉘어 owner, group, other의 권한을 나타낸다.

예를 들어 -rw-r--r--라면 일반 파일이고, 소유자는 읽기·쓰기 가능, 그룹은 읽기만 가능, 기타 사용자도 읽기만 가능하다. drwxr-xr-x라면 디렉토리이고, 소유자는 읽기·쓰기·진입 가능, 그룹과 기타 사용자는 읽기와 진입이 가능하지만 쓰기는 못 한다.

값권한파일에서의 의미디렉토리에서의 의미
4r내용 읽기목록 조회
2w내용 수정항목 생성·삭제·이름 변경
1x실행 파일 실행디렉토리 진입·경로 통과

숫자 권한은 read 4, write 2, execute 1을 더해 만든다. 7은 4+2+1이므로 rwx, 6은 4+2이므로 rw-, 5는 4+1이므로 r-x, 0은 아무 권한 없음이다. 세 자리 숫자 755는 owner 7, group 5, other 5라는 뜻이다. 강사는 이 부분이 정보보안기사 같은 시험에도 자주 나오므로 반드시 익혀야 한다고 강조한다.

새 파일과 디렉토리는 기본 권한에서 umask가 막는 권한을 제외해 만들어진다. 파일은 기본적으로 실행 권한이 빠진 666에서 시작하고, 디렉토리는 진입 권한이 필요하므로 777에서 시작한다. Ubuntu의 일반적인 umask 0002에서는 새 파일이 664, 새 디렉토리가 775처럼 만들어진다.

umask
touch hello2.txt
mkdir testdir
ls -l
umask 007
touch private.txt
mkdir private-dir
ls -l
umask 002

특수 권한 때문에 umask가 네 자리로 보일 수 있는데, 기초 단계에서는 뒤 세 자리 owner/group/other 마스크를 먼저 이해하면 된다. 앞쪽 한 자리는 SUID, SGID, sticky bit 같은 특수 권한과 연결된다.

SECTION 06

chmod, chown, chgrp로 권한과 소유권 바꾸기

chmod는 change mode의 약자이며, 파일이나 디렉토리의 권한 모드를 바꾼다. 두 가지 방식이 있다. 첫째는 숫자 모드다. chmod 777 hello.txt는 owner, group, other 모두에게 읽기·쓰기·실행 권한을 준다. chmod 700 hello.txt는 owner에게만 모든 권한을 주고 나머지는 모두 막는다.

chmod 777 hello.txt
chmod 700 hello.txt
chmod 755 script.sh
chmod 644 note.txt

둘째는 문자 모드다. 대상은 u(user/owner), g(group), o(other), a(all)로 지정하고, +는 추가, -는 제거, =는 정확히 설정을 의미한다.

chmod u+x hello.txt
chmod u-x hello.txt
chmod g+rw hello.txt
chmod g-rw hello.txt
chmod o+w hello.txt
chmod o-rx hello.txt

chmod u+x는 소유자에게 실행 권한을 더하고, chmod g-rw는 그룹에서 읽기와 쓰기 권한을 제거한다. chmod o+w처럼 other에게 쓰기 권한을 주면 다른 사용자도 파일을 바꿀 수 있으므로 실제 운영 환경에서는 매우 조심해야 한다.

sudo chown user2:user2 hello.txt
sudo chown user2 hello.txt
sudo chgrp user2 hello.txt
sudo chown user1:user1 hello.txt

chown은 change owner, chgrp는 change group이다. 실무에서는 chown user:group file처럼 한 번에 소유자와 그룹을 바꾸는 형태가 자주 쓰인다. 이 작업은 보통 관리자 권한이 필요하다. 아무 사용자가 파일 소유자를 마음대로 바꿀 수 있으면, 악성 파일을 남의 소유처럼 보이게 하거나 책임 추적을 흐릴 수 있기 때문이다.

강의는 특수 권한인 SUID, SGID, sticky bit도 짧게 소개한다. 이들은 일반 rwx 권한 앞에 붙는 추가 권한으로, 실행 시 소유자 권한을 빌리거나, 디렉토리 안의 그룹 상속을 조정하거나, 공유 디렉토리에서 삭제 권한을 제한하는 식으로 쓰인다. 기초반에서는 세부 활용까지 외우기보다, 나중에 앞자리 숫자와 연결해 다시 배울 내용이라고 이해하면 된다.

SECTION 07

실습: user1과 user2로 권한 차이 확인하기

강사는 VirtualBox 안의 글자가 작게 보일 수 있어 원격 터미널 환경으로 큰 글씨를 보여 주지만, 학생들은 Ubuntu GUI 터미널에서 같은 명령을 따라 하면 된다. 실습은 ls -l로 현재 파일들의 권한과 소유자, 그룹을 확인하는 것부터 시작한다.

ls
ls -l
touch hello2.txt
mkdir dir1
mkdir dir2
ls -l

기존 umask가 002라면 새 파일은 대체로 rw-rw-r--, 새 디렉토리는 rwxrwxr-x처럼 만들어진다. 이후 umask 007처럼 값을 바꾸고 새 파일과 디렉토리를 만들면 other 권한이 빠지는 것을 볼 수 있다. 이 실습은 umask가 “권한을 더하는 값”이 아니라 “기본 권한에서 빼는 마스크”라는 점을 확인하게 해 준다.

다음으로 sudo adduser user2를 실행해 두 번째 사용자를 만든다. 리눅스는 멀티유저 운영체제이므로 user1과 user2가 동시에 로그인할 수 있다. 강의에서는 한쪽 터미널은 user1, 다른 한쪽 터미널은 user2로 열어 같은 파일에 대한 접근 결과를 비교한다.

sudo adduser user2
su - user2
whoami
id
pwd
ls

user2의 id를 보면 user1처럼 sudo, adm, cdrom, netdev 같은 관리·장치 그룹에 들어 있지 않다. 또한 터미널에서 만든 계정은 GUI에서 처음 만든 계정과 달리 Downloads, Documents 같은 폴더가 아직 없을 수 있다. 이는 계정 생성 방식과 초기 설정 차이일 뿐, 이상 현상은 아니다.

echo hello > hello.txt
cat hello.txt
ls -l hello.txt

echo hello > hello.txt는 hello라는 문자열을 표준 출력으로 내보내고, > 리다이렉션으로 그 출력을 파일에 저장한다. 리다이렉션 자체는 뒤 배시 쉘 강의에서 더 자세히 다루지만, 여기서는 파일에 내용을 써서 권한 차이를 확인하는 데 사용한다.

user1이 만든 hello.txt에 other 읽기 권한이 있으면 user2는 cat hello.txt로 내용을 읽을 수 있다. 하지만 other 쓰기 권한이 없으면 echo hacker > hello.txt처럼 덮어쓰려 할 때 권한 거부가 난다. user1이 chmod o+w hello.txt로 other 쓰기 권한을 주면, 그때부터 user2도 파일 내용을 바꿀 수 있다. 이 장면이 파일 권한이 단순 표기가 아니라 실제 데이터 보호에 직접 연결된다는 핵심 실습이다.

chmod o+w hello.txt
# user2 터미널
echo hacker > hello.txt
cat hello.txt
# user1 터미널
chmod o-w hello.txt

디렉토리 권한은 파일 권한보다 더 헷갈리기 쉽다. 파일에서 x는 실행을 뜻하지만, 디렉토리에서 x는 그 디렉토리에 들어가거나 경로 중간을 통과할 수 있는 권한이다. r은 목록을 볼 수 있는 권한이고, w는 그 안에 항목을 만들거나 삭제하거나 이름을 바꿀 수 있는 권한이다.

chmod 771 dir2
# other는 dir2에 cd로 들어갈 수는 있지만 ls 목록 확인은 제한될 수 있다.
chmod 775 dir2
# other도 읽기와 진입이 가능해 목록을 볼 수 있다.
chmod 770 dir2
# other의 접근을 다시 막는다.

chmod 771 dir2는 other에게 execute만 주는 예다. 이 경우 경로를 알고 있으면 들어가거나 통과할 수 있지만, 목록을 읽는 것은 제한된다. chmod 775 dir2는 other에게 read와 execute를 주므로 목록 조회와 진입이 가능하다. 마지막으로 chmod 770 dir2 또는 chmod o-rx dir2를 적용하면 other의 접근을 다시 막을 수 있다.
