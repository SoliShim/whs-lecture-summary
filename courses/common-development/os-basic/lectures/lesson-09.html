SECTION 01

시즌2의 시작: 서버 프로그램을 직접 운영하기

이번 강의는 시즌2의 첫 강의다. 시즌1에서는 운영체제가 무엇인지, Linux와 Ubuntu가 무엇인지, 가상 머신에 Ubuntu를 설치하는 법, GUI와 CLI를 다루는 법, 파일과 디렉토리, 사용자 계정과 권한, 패키지 설치, 데몬 서비스 관리를 배웠다. 이제 그 위에 실제 서버 프로그램을 설치하고 운영하는 내용을 얹는다.

강사는 일부 설정이 온라인 강의만으로 따라 하기 어려울 수 있으므로, 영상을 멈추고 다시 재생하면서 직접 커맨드라인을 쳐 보라고 당부한다. 이 강의의 큰 축은 세 가지다. 첫째 웹 서버 구축, 둘째 파일 서버 구축, 셋째 데이터베이스 서버 구축이다. 세 영역은 모두 “패키지 설치 -> 설정 파일 확인과 수정 -> 서비스 재시작 -> 실제 접속 확인”이라는 공통 흐름을 가진다.

SECTION 02

웹 서버: Apache와 Nginx의 이름과 구조

웹 서버의 대표 프로그램은 Apache와 Nginx다. Apache는 더 오래된 역사와 전통을 가진 웹 서버다. Red Hat/Fedora 계열에서는 httpd라는 이름으로 부르고, Debian/Ubuntu 계열에서는 apache2라는 패키지와 서비스 이름으로 다룬다. Nginx는 Apache보다 늦게 나온 후발 주자지만, 최근 실무에서 매우 많이 쓰인다.

이론 설명은 Apache의 구조에서 출발한다. 많은 웹 서버 설정 구조가 Apache에서 먼저 정리되었고, Nginx도 유사한 방식으로 sites-available과 sites-enabled 구조를 사용한다. 다만 실습 시간은 제한되어 있으므로 실제 조작은 Nginx 중심으로 진행한다.

sudo apt install apache2
systemctl status apache2
sudo systemctl start apache2
sudo systemctl stop apache2
sudo systemctl restart apache2

Apache의 메인 설정은 /etc/apache2/apache2.conf에 있고, 사이트 설정은 /etc/apache2/sites-available/과 /etc/apache2/sites-enabled/ 구조로 관리한다. 웹 콘텐츠의 기본 위치는 /var/www/html이고, 로그는 /var/log/apache2/에 쌓인다. 설정 파일은 /etc, 변동 데이터와 로그는 /var 아래에 둔다는 앞 강의의 파일시스템 구조와 연결된다.

하나의 웹 서버 프로그램이 여러 사이트를 운영할 수 있다. 이것을 가상 호스트, virtual host라고 부른다. 예를 들어 site1, site2, site3 설정 파일을 sites-available에 만들어 두고, 실제 켤 사이트만 sites-enabled에 심볼릭 링크로 연결한다. 여기서 5강의 ln -s가 실제 서버 운영에 쓰인다. “설정 후보는 available에 보관하고, 활성화는 enabled의 링크로 결정한다”는 구조를 기억하면 된다.

Apache를 설치하고 서비스가 실행되면 브라우저에서 Apache2 Ubuntu Default Page를 볼 수 있다. 이것은 웹 서버가 포트를 열고, 요청을 받고, 기본 문서 루트의 HTML을 반환한다는 뜻이다.

SECTION 03

Nginx 설치와 설정 파일 읽기

Nginx도 Apache와 유사한 설정 구조를 가진다. 패키지는 sudo apt install nginx로 설치하고, 상태는 systemctl status nginx로 확인한다. 강사는 VirtualBox 안에서 직접 볼 수도 있고, Windows 호스트에서 VM의 호스트 전용 어댑터 IP로 접속할 수도 있다고 설명한다. 예시 화면의 IP는 강사의 환경 기준이며, 학생의 VM IP는 다를 수 있다.

sudo apt install nginx
systemctl status nginx
curl localhost
# VM 안 브라우저: http://127.0.0.1
# 호스트 PC 브라우저: http://<Ubuntu_VM_IP>

Nginx의 메인 설정 파일은 /etc/nginx/nginx.conf다. 이 파일에는 worker 설정, 로그 위치, gzip, SSL 관련 기본 설정, 그리고 다른 설정 파일을 불러오는 include 구문이 있다. 특히 include /etc/nginx/sites-enabled/*;가 중요하다. Nginx는 sites-available을 직접 읽는 것이 아니라, sites-enabled 안에 활성화된 링크를 읽는다.

경로의미
/etc/nginx/nginx.confNginx 메인 설정 파일
/etc/nginx/sites-available/default사이트별 실제 설정 파일
/etc/nginx/sites-enabled/default활성화된 설정을 가리키는 심볼릭 링크
/var/www/html기본 웹 콘텐츠 루트
/var/log/nginx/접속 로그와 오류 로그 위치
 
cd /etc/nginx
cat nginx.conf
ls -l sites-enabled
cat sites-available/default
cat /var/www/html/index.nginx-debian.html

sites-available/default의 server 블록에는 listen, root, index, server_name, location 설정이 들어 있다. listen 80은 80번 포트를 열겠다는 의미이고, root /var/www/html;은 요청한 파일을 찾을 기본 웹 루트다. index index.html index.htm ...은 디렉토리에 접근했을 때 기본으로 찾을 파일 이름이다.

SECTION 04

새 Nginx 가상 사이트 만들기와 포트 충돌 해결

새 사이트를 만들 때는 기존 default 설정을 복사해서 출발할 수 있다. 강사는 sites-available/default를 whiteschool이라는 이름으로 복사한 뒤, 주석을 지우고 필요한 설정만 남긴다. Vim에서 dd는 한 줄 삭제, 10dd는 10줄 삭제처럼 쓸 수 있다.

cd /etc/nginx/sites-available
sudo cp default whiteschool
sudo vi whiteschool
cd /etc/nginx/sites-enabled
sudo ln -s /etc/nginx/sites-available/whiteschool
ls -l
sudo systemctl restart nginx

여기서 바로 재시작하면 오류가 날 수 있다. 복사한 설정 파일도 listen 80이고, 기존 default 설정도 listen 80이면 하나의 서버에서 같은 IP와 같은 포트를 두 설정이 동시에 잡으려 하기 때문이다. systemctl status nginx나 journalctl -u nginx를 보면 “80 포트를 이미 사용 중”이라는 bind 오류를 확인할 수 있다.

sudo nginx -t
sudo vi /etc/nginx/sites-available/whiteschool
# listen 80; 을 listen 8000; 등으로 변경
sudo nginx -t
sudo systemctl restart nginx
curl localhost:8000

sudo nginx -t는 Nginx 설정 문법과 기본적인 충돌을 재시작 전에 검사한다. 설정에 문제가 없으면 syntax is ok, test is successful처럼 나온다. 오류가 있으면 어느 파일의 몇 번째 줄인지 알려 주므로, 재시작 전에 반드시 확인하는 습관을 들여야 한다.

sudo mkdir -p /var/www/html/whiteschool
echo '<h1>Hello, WHS</h1>' | sudo tee /var/www/html/whiteschool/index.html
sudo vi /etc/nginx/sites-available/whiteschool
# root /var/www/html/whiteschool; 로 변경
sudo nginx -t
sudo systemctl restart nginx
curl localhost
curl localhost:8000

80번 포트의 기본 사이트는 기존 Nginx 기본 페이지를 보여 주고, 8000번 포트의 새 사이트는 Hello, WHS 같은 별도 콘텐츠를 보여 줄 수 있다. 이 실습은 하나의 서버 안에서 여러 사이트를 설정 파일과 포트로 나누어 운영할 수 있음을 보여 준다.

SECTION 05

FTP 파일 서버와 vsftpd

웹 서버 다음은 파일 서버다. 강의에서는 대표적인 FTP 서버로 vsftpd를 사용한다. FTP는 과거 문서, 유틸리티, 배포판 이미지, MP3, 영상 파일을 업로드·다운로드하는 데 많이 쓰였던 파일 전송 프로토콜이다. 지금은 SFTP, 클라우드 스토리지, 웹 기반 파일 공유가 더 많이 쓰이지만, 서버와 클라이언트, 계정, 권한, 디렉토리 제한을 배우는 데 좋은 예제다.

웹 서버처럼 설정 파일이 항상 폴더 구조를 갖는 것은 아니다. Apache와 Nginx는 설정이 많아 /etc/apache2/, /etc/nginx/ 아래 여러 파일과 폴더를 쓰지만, vsftpd는 기본 설정이 비교적 단순해 /etc/vsftpd.conf 단일 파일을 중심으로 관리한다. 설정 파일 구조는 Ubuntu가 임의로 정한다기보다 각 서버 프로그램을 만든 쪽의 설계에 따라 달라진다.

FTP 클라이언트는 Ubuntu와 Windows에 기본 CLI 명령으로 들어 있을 수 있고, GUI 클라이언트로 FileZilla 같은 서드파티 도구도 있다. Ubuntu 내부 FTP CLI에서는 한글 파일명이 깨질 수 있고, GUI 클라이언트에서도 설정에 따라 깨질 수 있다. 이런 경우 서버의 UTF-8 관련 설정을 조정하고 서비스를 재시작해 확인한다.

vsftpd의 주요 설정은 다음처럼 이해한다. anonymous_enable은 익명 접속 허용 여부, local_enable은 로컬 OS 계정 접속 허용 여부, write_enable은 업로드·삭제 같은 쓰기 작업 허용 여부, chroot_local_user는 사용자를 홈 디렉토리 밖으로 나가지 못하게 가두는 설정이다. 쓰기 권한은 업로드뿐 아니라 삭제와 변경까지 가능하게 하므로 기본값이 꺼져 있는 이유를 이해해야 한다.

SECTION 06

FTP 실습: 접속, 다운로드, 업로드, chroot

실습에서는 먼저 vsftpd가 설치되어 있지 않음을 systemctl status vsftpd로 확인한 뒤 설치한다. 설치 중 systemd에 서비스가 등록되고, 설치 후에는 active running 상태가 된다.

systemctl status vsftpd
sudo apt install vsftpd
systemctl status vsftpd
ftp 127.0.0.1

Ubuntu 안에서 ftp 127.0.0.1로 접속하면 현재 로그인한 사용자 user1이 기본 사용자명처럼 들어갈 수 있다. 비밀번호를 입력하면 user1의 홈 디렉토리를 볼 수 있고, ls, pwd, get, put, cd 같은 FTP 명령을 사용할 수 있다. 로컬에서 만든 hello.txt가 FTP 접속에서도 보이는 것을 통해, FTP가 OS 사용자 홈 디렉토리를 보여 주고 있음을 확인한다.

# Windows 또는 다른 클라이언트에서
ftp <Ubuntu_VM_IP>
user1
get hello.txt
quit
type hello.txt

Windows에서 접속할 때 127.0.0.1은 Windows 자신을 의미하므로 Ubuntu VM에 접속되지 않는다. VM의 호스트 전용 어댑터 IP로 접속해야 한다. Ubuntu 안에서 Ubuntu 자신으로 접속할 때와 달리, Windows 클라이언트에서는 사용자명을 직접 입력해야 한다.

기본 설정에서는 업로드가 막혀 있다. Windows에서 메모장으로 myFile.txt를 만들고 FTP에서 put myFile.txt를 실행하면 permission denied가 날 수 있다. 이때 서버의 /etc/vsftpd.conf에서 쓰기 권한을 켜야 한다.

sudo vi /etc/vsftpd.conf
# /etc/vsftpd.conf 예시
write_enable=YES
sudo systemctl restart vsftpd
systemctl status vsftpd
ftp <Ubuntu_VM_IP>
put myFile.txt

설정 파일을 바꿨으면 항상 서비스를 재시작해야 한다. 재시작 후 다시 접속하면 put 업로드가 성공하고, 서버 홈 디렉토리에 파일이 생긴 것을 확인할 수 있다.

sudo vi /etc/vsftpd.conf
chroot_local_user=YES
allow_writeable_chroot=YES
sudo systemctl restart vsftpd
ftp <Ubuntu_VM_IP>
cd ..
pwd

처음 상태에서는 FTP 로그인 후 cd ..로 상위 디렉토리로 올라가 시스템의 여러 파일을 볼 수 있다. 이것은 운영 환경에서는 위험하다. chroot_local_user=YES를 켜면 사용자의 홈 디렉토리를 FTP 세션의 루트처럼 만들어 상위로 벗어나지 못하게 할 수 있다. 일부 vsftpd 버전에서는 홈 디렉토리가 쓰기 가능할 때 로그인이 거부될 수 있어 allow_writeable_chroot=YES가 함께 필요하다.

최종적으로 다운로드, 업로드, chroot 제한이 모두 동작하는지 확인한다. 이 강의에서는 모든 FTP 설정을 다 실습하지는 않지만, 익명 접속 허용, 특정 사용자 허용/거부, 화이트리스트/블랙리스트 같은 접근 제어도 설정 파일을 통해 더 확장할 수 있다고 설명한다.

SECTION 07

데이터베이스 서버: MySQL 설치와 최초 접속

마지막은 데이터베이스 서버다. DB 서버에는 MySQL, MariaDB, MSSQL, PostgreSQL 등 여러 종류가 있다. 이 강의에서는 가장 대표적인 MySQL을 사용한다. 데이터베이스 수업이 아니므로 SQL 문법을 깊게 다루지는 않고, 리눅스 서버에 DB 서버를 설치하고 서비스 상태를 확인한 뒤 기본 사용자와 권한을 만드는 흐름만 익힌다.

sudo apt install mysql-server
sudo mysql_secure_installation
systemctl status mysql
sudo mysql

mysql-server 패키지를 설치하면 서버뿐 아니라 MySQL 클라이언트와 여러 의존 패키지가 함께 설치된다. 설치 중 패키지 목록이 오래되어 다운로드가 실패하면 앞 강의처럼 sudo apt update 후 다시 설치한다. 설치가 끝날 때 systemd에 MySQL 서비스가 등록되는 심볼릭 링크 메시지도 볼 수 있다.

mysql_secure_installation은 암호 복잡도, root 암호, 익명 사용자, 원격 root 로그인, 테스트 DB 같은 보안 관련 기본 설정을 조정하는 도구다. 강의에서는 복잡한 정책 없이 기본 실습을 진행하지만, 운영 환경에서는 반드시 보안 설정을 신중히 적용해야 한다. Ubuntu 18.04 이후 흐름에서는 초기 관리 접속을 sudo mysql로 수행한다. 과거 자료에서 보이는 mysql -u root 방식은 버전과 배포판에 따라 다를 수 있다.

SECTION 08

MySQL 내부 사용자, 권한, 테이블 실습

FTP는 Ubuntu의 OS 사용자 계정으로 로그인할 수 있었다. 하지만 MySQL은 DB 내부 사용자와 권한을 따로 관리한다. 따라서 데이터베이스를 만들고, DB 사용자를 만들고, 그 사용자가 어느 위치에서 접속할 수 있는지와 어떤 데이터베이스에 권한을 가질지를 별도로 지정해야 한다.

sudo mysql

SHOW DATABASES;
CREATE DATABASE whiteschool;
CREATE USER 'user1'@'localhost' IDENTIFIED BY 'qwe123';
GRANT ALL PRIVILEGES ON whiteschool.* TO 'user1'@'localhost';
FLUSH PRIVILEGES;
SELECT user, host FROM mysql.user;
EXIT;

mysql -h localhost -u user1 -p
USE whiteschool;

SHOW DATABASES;는 현재 보이는 데이터베이스 목록을 출력한다. CREATE DATABASE whiteschool;은 새 데이터베이스를 만든다. CREATE USER 'user1'@'localhost'는 localhost에서 접속할 수 있는 DB 사용자 user1을 만든다는 뜻이다. 여기서 user1은 Ubuntu OS 사용자 user1과 이름이 같을 수는 있지만 별개의 DB 사용자다.

GRANT ALL PRIVILEGES ON whiteschool.* TO 'user1'@'localhost';는 user1에게 whiteschool 데이터베이스의 모든 테이블에 대한 권한을 준다. FLUSH PRIVILEGES;는 권한 변경을 반영한다. 그 후 mysql -h localhost -u user1 -p로 새 DB 사용자로 접속해 USE whiteschool;을 실행하면, 해당 데이터베이스 안에서 작업할 수 있다.

CREATE TABLE students (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50)
);
INSERT INTO students (name) VALUES ('user1');
SHOW TABLES;
SELECT * FROM students;

강의는 데이터베이스 문법 자체를 깊게 다루지는 않는다. 중요한 점은 리눅스 서버 운영 관점에서 MySQL도 하나의 서버 프로그램이며, 패키지 설치, 서비스 상태 확인, 초기 접속, 내부 사용자 생성, 권한 부여, 접속 테스트라는 운영 흐름을 가진다는 것이다.

네, 좀 더 최근 자료를 보면 또 이제 최근에는 이렇게 이제 그 클라우드 플레어라고 불리는 네, 이제 그런 CDN 서비스 업체라고 불리는데 네, 이런 서비스를 통해서 좀 더 편하게 쉽게 구축할 수 있는 이런 또 이제 웹 서버가 최근에 많은 시장 점유율을 확보하고 있는 것을 알 수 있습니다. 네, 그래서 오늘 제가 이제 이론 파트에서는 아파치 서버한테 또 이제 역사와 전통을 자랑하는 네, 기본적으로 이제 아파치로 모든 설정들이 출발을 했고 그 후에 후발 주자로 엔지넥스가 생겨나면서 네, 아파치의 설정들을 그대로 이렇게 계승받아서 뭐, 뭐 소거를 사용하자면 좀 이렇게 벗겨서 만든 거다 보니까 그래서 일단 아파치에 대한 설정을 먼저 배워보도록 할 예정이고요. 네, 그 후로 이제 실스 파트에서는 네, 저희가 시간이 짧기 때문에 엔지넥스를 기반으로만 실습을 진행할 예정입니다. 자, 그래서 아파치에 대한 웹 서버를 설치하려고 하면은 네, 우리가 앞서 배운 것과 동일한 명령어가 되겠습니다. 네, sudo apt install apache2라는 명령어를 통해서 우분투에다가 아파치 웹 서버를 설치할 수가 있습니다.
